二、4.1 设备管理基础

2.1 设备管理概述

一句话理解

I/O 设备是计算机与外界交换信息的桥梁，而设备管理就是操作系统对这些桥梁的统一调度与控制。

2.1.1 什么是 I/O 设备

I/O 设备，也叫：

  • 输入输出设备
  • 外围设备
  • 外部设备
  • 外设

它们的作用是：

  • 与用户交换信息
  • 与其他计算机通信
  • 长期保存数据

I/O 操作的定义

I/O 操作 = 内存和外设之间的信息传送操作

注意：

  • CPU 直接处理的是内存中的数据
  • 外设速度通常远低于 CPU
  • 所以 I/O 往往是整个系统性能瓶颈之一

2.1.2 I/O 设备分类

1）按信息传输方向分类

类别 含义 例子
输入设备 向计算机输入信息 键盘、鼠标、扫描仪
输出设备 输出计算结果 显示器、打印机
输入输出设备 既能输入又能输出 磁盘、网卡

2）按交互功能分类

类别 含义 例子
人机交互设备 用户与计算机交互 键盘、鼠标、显示器
存储设备 存储和检索信息 磁盘、光盘
机机通信设备 计算机之间通信 网卡、调制解调器

3）按设备管理视角分类

这是操作系统最常用、最重要的分类。

类别 特点 例子
字符设备 以字符为单位传输，通常顺序访问 键盘、鼠标、显示器、打印机
块设备 以固定大小的数据块传输，可随机访问 磁盘
网络设备 用于通信，抽象方式灵活 网卡

2.1.3 为什么设备管理很重要

设备管理会影响：

  • 计算机系统的通用性
  • 系统的可扩充性
  • 主机与设备的并行程度
  • 整体吞吐量
  • 系统性价比

根本矛盾

CPU 快，设备慢；设备之间速度也不一样。

例如：

  • CPU：纳秒级
  • 内存：几十到上百纳秒
  • 磁盘：毫秒级
  • 键盘输入：人类速度级

这导致如果管理不好，CPU 会大量空等。

2.1.4 设备管理的目标

目标 1：提高设备利用率与系统并行性

让：

  • CPU 工作时设备也能工作
  • 设备工作时 CPU 不要傻等

目标 2：屏蔽设备差异

不同设备：

  • 接口不同
  • 命令不同
  • 速度不同
  • 数据单位不同

操作系统要给用户和程序提供统一接口。

目标 3：实现设备独立性

应用程序最好不直接依赖某台具体物理设备，而是依赖逻辑设备。

目标 4：将设备抽象化

常见抽象方式：

  • 裸设备：直接作为原始设备访问
  • 设备文件：把设备看成文件来操作

2.1.5 设备管理的功能

功能 解释
设备中断处理 响应 I/O 完成、错误等中断
缓冲区管理 利用缓冲改善速度不匹配
设备分配和回收 把设备分给进程，用完回收
设备驱动调度 按合理顺序执行 I/O 请求
虚拟设备实现 用共享设备模拟独占设备，如 SPOOLing

2.1.6 设备管理的实现层次

设备管理不是只靠软件，也不是只靠硬件，而是硬件和软件共同完成。

flowchart TD
    A[用户程序] --> B[用户空间I/O软件]
    B --> C[设备无关I/O软件]
    C --> D[设备驱动程序]
    D --> E[中断处理程序]
    E --> F[设备控制器]
    F --> G[I/O设备]

典型例子

例：打印一份文档

实际过程并不是“程序直接把内容送到打印机”：

  1. 应用程序发起打印请求
  2. 操作系统通过统一 I/O 接口接收请求
  3. 设备无关软件进行命名、权限检查、缓冲等
  4. 打印机驱动把逻辑请求翻译成打印机可识别命令
  5. 控制器执行实际传输
  6. 打印完成后产生中断
  7. OS 唤醒等待进程或继续后续任务

高频考点

  • I/O 设备分类
  • 设备管理目标
  • 设备管理基本功能
  • 字符设备与块设备区别

易错点

  1. 字符设备不等于输入设备

    • 显示器是输出设备，但也常归入字符设备

  2. 块设备的“块”是管理单位，不一定等于文件逻辑记录

    • 文件逻辑记录大小和物理块大小可不同

  3. 设备管理目标不只是“提高速度”

    • 还包括统一接口、设备独立性、可靠性等

简答题模板

题目：简述设备管理的目标

答：
设备管理的目标主要包括：

  1. 解决 CPU 与 I/O 设备速度不匹配的问题，使主机与外设尽可能并行工作，提高设备利用率和系统吞吐量；
  2. 屏蔽设备的物理细节和操作差异，提供统一、友好的 I/O 接口；
  3. 实现设备独立性，使用户程序不依赖具体物理设备；
  4. 通过缓冲、调度、分配和虚拟设备等技术，提高系统效率和可靠性。

  • 分类：输入、输出、输入输出；字符、块、网络
  • 目标：并行、高效、统一、独立
  • 功能：中断、缓冲、分配、调度、虚拟设备

2.2 I/O 控制方式

一句话理解

I/O 控制方式描述的是“CPU 到底怎样指挥设备完成输入输出”。

2.2.1 设备控制器

CPU 一般不直接操作机械设备本体，而是通过设备控制器间接控制设备。

设备控制器别名

  • 设备适配器
  • I/O 控制器
  • I/O 接口
  • I/O 模块

为什么需要设备控制器

因为设备的机械部分和电子控制部分通常分开设计：

  • 机械部分：真正执行读写、打印、扫描
  • 电子部分：负责与主机通信、状态反馈、控制逻辑

控制器的核心功能

功能 说明
接口功能 作为 CPU 与设备之间的接口
命令接收 接收并识别 CPU/通道发来的命令
数据交换 完成数据传输
状态记录 记录设备状态供 CPU 查询
设备识别 多设备连接时识别设备地址

2.2.2 设备控制器组成

课件给出的结构可概括为：

  • 数据缓冲寄存器
  • 状态/控制寄存器
  • 地址译码与 I/O 控制逻辑
  • 主机侧总线接口
  • 设备侧接口控制逻辑
flowchart LR
    CPU <--地址/数据/控制线--> C[设备控制器]
    C --> R1[数据缓冲寄存器]
    C --> R2[状态/控制寄存器]
    C --> L[地址译码与控制逻辑]
    C --> D1[外设接口1]
    C --> D2[外设接口2]
    D1 --> P1[设备1]
    D2 --> P2[设备2]

2.2.3 三种基本 I/O 控制方式

课件重点是：

  1. 轮询方式
  2. 中断方式
  3. DMA 方式

此外还补充了 I/O 通道方式

（一）轮询方式

定义

CPU 发出 I/O 命令后，不断反复检查设备状态，直到设备就绪，再由 CPU 自己完成数据交换。

流程

flowchart TD
    A[CPU发出I/O命令] --> B[查询控制器状态]
    B --> C{设备是否就绪}
    C -- 否 --> B
    C -- 是 --> D[CPU执行数据读写]
    D --> E[检查结束状态]
    E --> F[结束]

特点

  • CPU 要一直等待设备
  • CPU 参与数据交换
  • 实现简单
  • 效率低，浪费 CPU

适用场景

  • 简单系统
  • 设备少、I/O 不频繁
  • 硬件条件有限的场合

（二）中断方式

定义

CPU 发出 I/O 命令后，不再傻等，而是继续执行其他工作；设备准备好或操作完成后，向 CPU 发出中断请求，CPU 再转去处理中断。

流程

flowchart TD
    A[CPU发出I/O命令] --> B[CPU继续执行其他指令]
    B --> C[设备准备就绪/完成]
    C --> D[设备发中断]
    D --> E[CPU响应中断]
    E --> F[中断处理程序传送数据/处理结果]
    F --> G[唤醒相关进程或结束]

特点

  • CPU 不等待设备就绪
  • 但 CPU 仍参与数据交换
  • 比轮询高效
  • 中断频繁时有开销

和轮询的本质区别

  • 轮询：主动问
  • 中断：被动等通知

（三）DMA 方式

定义

DMA（Direct Memory Access，直接存储器访问）方式中，DMA 控制器代替 CPU 直接控制主存与设备之间的数据传送。

为什么 DMA 更高效

大量数据传输时，如果还让 CPU 一个字一个字搬运，会很浪费。DMA 允许：

  • CPU 只在开始时设置参数
  • DMA 自己传送整块数据
  • 结束时再中断通知 CPU

流程

flowchart TD
    A[CPU设置DMA参数并启动I/O] --> B[CPU继续执行其他任务]
    B --> C[DMA控制器控制设备与内存传输]
    C --> D[全部数据传输完成]
    D --> E[DMA向CPU发中断]
    E --> F[CPU进行结束处理]

DMA 控制器通常包含

  • 数据计数器
  • 数据寄存器
  • 地址寄存器
  • 控制逻辑

特点

项目 DMA方式
CPU等待设备就绪 不需要
CPU参与数据交换 不参与主存与设备间的数据搬运
适合 大批量高速数据传输
效率 高

2.2.4 周期窃取

概念

DMA 和 CPU 都要访问内存时，DMA 会“借用”或“窃取”若干主存周期。

这叫：

  • 周期窃取
  • Cycle Stealing

理解

不是 DMA 一旦工作就把内存长期霸占，而是：

  • CPU 暂时让出少量内存总线周期
  • DMA 迅速传一部分数据
  • 然后 CPU 继续

所以对 CPU 影响通常不大。

2.2.5 I/O 通道方式

这是比 DMA 更高级的一种 I/O 控制方式。

定义

I/O 通道是一种专门负责 I/O 的处理机构，也称：

  • 通道控制器
  • I/O 处理器

核心思想

CPU 不再直接执行 I/O 指令，而是：

  1. 在内存中组织通道程序
  2. 启动指定通道
  3. 通道独立控制设备完成 I/O
  4. 完成后中断 CPU

四级连接结构

flowchart LR
    CPU --> Channel[I/O通道]
    Channel --> Controller[设备控制器]
    Controller --> Device[设备]

通道方式优势

  • CPU 与 I/O 并行程度更高
  • 更适合大型系统
  • 减轻 CPU 负担

2.2.6 四种 I/O 控制方式对比

方式 CPU是否等待设备 CPU是否参与数据交换 硬件复杂度 适合场景
轮询 是 是 低 简单系统
中断 否 是 中 一般 I/O
DMA 否 否（主存与设备间） 较高 大批量高速传输
通道 否 基本不参与具体 I/O 高 大中型系统

高频考点

  • 轮询、中断、DMA 的区别
  • DMA 的优点
  • 周期窃取含义
  • 通道方式工作流程

易错点

  1. 中断方式中 CPU 不等于完全不参与数据传输

    • 设备就绪后，CPU 要处理中断并参与相关数据处理

  2. DMA 不是不需要 CPU

    • CPU 需要初始化 DMA、启动 I/O、处理完成中断

  3. 通道不是设备控制器

    • 通道层次高于控制器，是更独立的 I/O 处理机构

简答题模板

题目：比较轮询方式、中断方式和 DMA 方式

答：
轮询方式中，CPU 发出 I/O 命令后不断查询设备状态，既等待设备就绪，又参与数据交换，因此实现简单但效率最低。
中断方式中，CPU 发出命令后可继续执行其他程序，待设备完成后通过中断通知 CPU，因此 CPU 不必等待设备就绪，但仍需处理中断并参与数据交换。
DMA 方式中，CPU 只在 I/O 开始和结束时参与，数据在设备与主存之间由 DMA 控制器直接传送，因此效率更高，特别适合大批量数据传输。

考前速背版

  • 轮询：等设备、搬数据
  • 中断：不等设备、还要搬数据
  • DMA：不等设备、也不搬数据
  • 通道：专门 I/O 处理器，更高级

2.3 总线与 I/O

一句话理解

总线是 CPU、内存和 I/O 设备共享通信的公共通道。

2.3.1 为什么需要总线

I/O 速度差异巨大：

  • CPU 很快
  • 主存较快
  • 外设很多且速度跨度极大

如果每个设备都直接和 CPU 独立连接：

  • 接线复杂
  • 扩展困难
  • 成本高

所以用总线来统一连接。

2.3.2 总线的作用

  • 连接 CPU、内存、I/O 模块
  • 提供统一数据通路
  • 方便扩展设备
  • 促进并行处理

2.3.3 单总线结构

结构

CPU、主存、I/O 模块全部连接到一条总线上。

flowchart LR
    CPU --> Bus[单总线]
    MEM[主存] --> Bus
    IO1[I/O模块1] --> Bus
    IO2[I/O模块2] --> Bus
    IO3[I/O模块3] --> Bus

优点

  • 结构简单
  • 容易扩充

缺点

  • 总线共享，竞争严重
  • 慢设备会拖慢总线
  • 设备多时压力大，时延增加

2.3.4 三级总线模型

课件强调：

  • 主存和 Cache 通过主存总线连接
  • 主存总线与扩展总线之间通过接口缓冲
  • I/O 设备挂在扩展总线上

目的

把：

  • 处理器-内存高速活动
  • I/O 低速活动

分离开来。

flowchart TD
    CPU --> MBus[主存总线]
    Cache --> MBus
    MEM[主存] --> MBus
    MBus --> IF[扩展总线接口]
    IF --> EBus[扩展总线]
    EBus --> IO1[I/O设备1]
    EBus --> IO2[I/O设备2]
    EBus --> IO3[I/O设备3]

优点

  • 支持更多设备
  • 减少 I/O 对内存/CPU 的干扰
  • 提高系统整体性能

缺点

  • 结构更复杂
  • 对设备速率差异极大的情况仍需更多优化

高频考点

  • 单总线与多总线比较
  • 总线的意义
  • 总线为什么能提高系统效率

易错点

  • 总线不是“提高某个设备自身速度”，而是改善整个系统的连接与并行组织
  • 总线本身也可能成为瓶颈

简答题模板

题目：说明单总线结构的优缺点

答：
单总线结构是指 CPU、主存和各 I/O 模块都连接在同一条总线上。其优点是结构简单、成本较低、易于扩充；缺点是所有部件共享同一总线，设备较多时总线竞争严重，慢速设备会占用带宽，导致传输延迟增加，系统性能下降。

考前速背版

  • 总线：公共通信通路
  • 单总线：简单但竞争大
  • 多级总线：分层、缓冲、支持更多设备

三、4.2 设备管理软件

3.1 I/O 软件的实现层次

一句话理解

I/O 软件分层的目的，是让高层看到统一接口，让低层处理硬件细节。

3.1.1 I/O 软件设计目标

课件给出两个核心目标：

  1. 高效率
  2. 通用性

高效率

重点是改善设备使用效率，尤其是磁盘 I/O。

通用性

用统一标准管理各种设备，让用户不必了解硬件差异。

3.1.2 设计时要考虑的问题

问题 含义
设备无关性 程序访问设备时不依赖具体硬件
出错处理 能在低层处理的错误尽量不暴露给高层
同步/异步传输 支持阻塞 I/O 与中断驱动 I/O
缓冲技术 用缓冲解决速度不匹配、提高吞吐量

3.1.3 I/O 软件层次结构

课件中的标准层次如下：

  1. 用户空间的 I/O 软件
  2. 独立于设备的 I/O 软件
  3. 设备驱动程序
  4. 中断处理程序
  5. I/O 硬件
flowchart TD
    A[用户空间I/O软件] --> B[设备无关I/O软件]
    B --> C[设备驱动程序]
    C --> D[中断处理程序]
    D --> E[I/O硬件]

3.1.4 各层职责详解

（1）用户空间 I/O 软件

包括：

  • 库函数
  • I/O 格式化
  • SPOOLing 软件
  • 一些运行在内核外的 I/O 支持

作用

给应用程序提供直接可用的接口，如：

  • printf
  • scanf
  • 文件流读写
  • 格式转换

（2）设备无关 I/O 软件

这是很重要的一层。

作用

处理适用于所有设备的通用功能：

  • 设备命名
  • 设备保护
  • 提供统一数据单位
  • 缓冲管理
  • 分配与回收
  • 状态跟踪
  • 统一错误处理

为什么叫“设备无关”

因为这层不关心具体是打印机还是磁盘，它只负责通用规则。

（3）设备驱动程序

驱动程序是最接近设备的软件层。

核心任务

  • 接收上层的逻辑 I/O 请求
  • 转换成具体设备可执行的物理操作
  • 初始化设备
  • 启动数据传输
  • 监督设备工作
  • 必要时进行纠错

直观理解

驱动程序相当于：

“操作系统和具体硬件之间的翻译官 + 操作员”

（4）中断处理程序

位置

位于 OS 底层，与硬件关系最密切。

功能

  • 检查状态寄存器，判断中断原因
  • 正常完成则唤醒等待进程
  • 出错则上报并请求处理
  • 如果队列中还有请求，则启动下一次 I/O

3.1.5 为什么必须分层

好处

好处 解释
降低复杂度 每层只管自己职责
便于移植 换设备通常只换驱动层
统一接口 上层程序无需关心硬件差异
易于维护 问题定位更清晰

典型例子

例：程序读取磁盘文件

  • I/O 软件层次结构
  • 设备无关软件与驱动程序区别
  • 分层设计目标
  • 同步与异步 I/O 的区别

易错点

  1. 驱动程序不是全部 I/O 软件

    • 它只是其中一层

  2. 设备无关软件不是“不需要设备”

    • 它是“不依赖具体某种设备”

  3. 同步/异步 与 阻塞/非阻塞有联系但不完全等价

    • 课程中通常可近似按阻塞/非阻塞理解，但更严格地说二者不是完全同义

简答题模板

题目：简述 I/O 软件的层次结构及各层作用

答：
I/O 软件通常采用分层结构，自上而下包括用户空间 I/O 软件、设备无关 I/O 软件、设备驱动程序、中断处理程序和 I/O 硬件。
用户空间 I/O 软件向应用程序提供库函数、格式化 I/O 和虚拟设备服务；设备无关 I/O 软件负责设备命名、保护、缓冲、分配、错误处理等通用功能；设备驱动程序把上层逻辑 I/O 请求转换为具体物理设备操作；中断处理程序负责响应 I/O 中断、检查设备状态、唤醒等待进程并处理后续请求；I/O 硬件最终执行实际的数据传输。
这种分层结构有利于屏蔽硬件差异、提高系统通用性和可维护性。

考前速背版

  • 层次：用户空间 → 设备无关 → 驱动 → 中断 → 硬件
  • 设备无关层：命名、保护、缓冲、分配、错误处理
  • 驱动层：把逻辑请求翻译成物理操作
  • 中断层：处理完成/出错通知

3.2 I/O 软件的实现

一句话理解

I/O 请求从发出到完成，要经过“检查—排队—启动—中断—唤醒”这一整套流程。

3.2.1 I/O 操作执行步骤

课件给出了一条很标准的执行路径，这部分很适合考试简答。

Step 1：应用进程发出 I/O 请求

例如对已打开文件执行读操作。

Step 2：设备无关软件检查参数和缓冲

  • 参数是否合法
  • 权限是否允许
  • 缓冲中是否已有所需数据

如果缓存中已经有数据，可以直接返回，避免物理 I/O。

Step 3：若缓冲没有数据，则进入物理 I/O

包括：

  • 逻辑设备名转换为物理设备名
  • 检查访问权限
  • 把请求排入 I/O 队列
  • 阻塞请求进程，等待完成

Step 4：驱动程序启动设备

  • 分配缓冲区
  • 设置控制寄存器
  • 或建立 DMA 传输
  • 启动设备

Step 5：设备控制器执行操作

真正和设备交互，完成读写。

Step 6：完成后产生中断

DMA 或设备控制器在一块数据完成后发中断。

Step 7：CPU 响应中断

转到磁盘/设备中断处理程序。

Step 8：进程被唤醒并继续执行

当该进程再次获得 CPU，就从系统调用断点恢复运行。

整体流程图

flowchart TD
    A[进程发出read/write] --> B[设备无关软件检查参数]
    B --> C{缓冲中是否已有数据}
    C -- 是 --> D[直接从缓冲返回]
    C -- 否 --> E[逻辑名转物理名/检查权限/排队]
    E --> F[阻塞进程]
    F --> G[驱动程序启动设备或DMA]
    G --> H[设备执行I/O]
    H --> I[产生I/O完成中断]
    I --> J[中断处理程序处理结果]
    J --> K[唤醒等待进程]
    K --> L[进程重新调度并继续执行]

3.2.2 中断处理程序

功能总结

  1. 判断中断原因
  2. 处理正常完成或错误
  3. 唤醒等待进程
  4. 启动下一个等待中的 I/O 请求

为什么中断处理程序必须尽量精简

因为它运行在内核底层，通常需要：

  • 快速响应
  • 减少中断关闭时间
  • 防止影响整个系统实时性

3.2.3 设备驱动程序

本质

设备驱动程序包含与设备密切相关的所有控制代码。

三大核心功能

  1. 设备初始化
  2. 启动和执行 I/O
  3. 设备相关中断处理

驱动分层思想

课件提到驱动也可以分层：

  • 高层处理“设备类”
  • 低层处理“具体硬件”

这种做法便于：

  • 结构清晰
  • 代码复用
  • 驱动移植

但会带来额外系统开销。

3.2.4 独立于设备的软件

再次强调一下它和驱动的区别：

比较项 设备无关软件 设备驱动程序
是否依赖具体设备 否 是
功能 命名、保护、缓冲、分配等共性工作 寄存器设置、启动设备、控制传输
面向对象 所有设备 某一类或某一种设备

3.2.5 用户空间 I/O 软件

包括

  • I/O 库函数
  • 格式化输入输出
  • SPOOLing 类软件
  • 一些仿真/重定向功能

为什么放在用户空间

因为不是所有 I/O 支持都必须进入内核，部分内容：

  • 放在用户空间更灵活
  • 易于扩展
  • 可减少内核复杂度

高频考点

  • I/O 请求执行流程
  • 中断处理程序功能
  • 设备驱动程序功能
  • 设备无关软件与驱动的区别

易错点

  • 唤醒进程不等于马上执行

    • 只是变为就绪态，仍要等待调度

  • 驱动程序不是简单“收发数据”

    • 还要负责初始化、状态监督、错误处理

简答题模板

题目：简述一次读操作的实现过程

答：
当进程发出读请求后，首先由设备无关 I/O 软件检查参数和权限，并检查缓冲区中是否已有所需数据；若缓冲中命中，则直接返回数据。若未命中，则将逻辑设备名转换为物理设备名，把请求排队并阻塞该进程。随后设备驱动程序初始化设备、设置控制寄存器或 DMA 参数并启动 I/O。设备控制器完成数据传输后产生中断，CPU 转入中断处理程序，对完成状态或错误进行处理，并唤醒等待该 I/O 的进程。进程再次被调度时，从系统调用断点继续执行。

考前速背版

  • I/O 流程：请求 → 检查 → 排队/阻塞 → 启动设备 → 中断 → 唤醒
  • 中断程序：查原因、报错、唤醒、启动下一个
  • 驱动程序：初始化、启动传输、设备相关控制

3.3 I/O 缓冲

一句话理解

缓冲就是在内存中搭一块“中转站”，用来缓和 CPU 和设备之间的速度差。

3.3.1 为什么需要缓冲

课件列出的缓冲目的非常重要，几乎可以直接背。

缓冲的作用

  1. 解决 CPU 与设备速度不匹配
  2. 协调逻辑记录与物理记录大小不一致
  3. 提高 CPU 和设备并行性
  4. 减少中断次数
  5. 放宽 CPU 对中断响应时间的要求

3.3.2 什么是 I/O 缓冲区

I/O 缓冲区是内存中的专门区域，用于暂存 I/O 数据。

写操作

  • 先把数据写到缓冲区
  • 缓冲满或满足条件后，再统一写到设备

读操作

  • 先把设备上的物理记录读到缓冲区
  • 再把当前所需部分送给进程

3.3.3 单缓冲

原理

系统只设置一个缓冲区。

输入时

  1. 设备把数据送到缓冲区
  2. 再由缓冲区送到用户区
  3. 用户进程处理数据
  4. 重复以上过程

特点

  • 比无缓冲好
  • 但并行程度有限
  • 某些阶段仍需等待
flowchart LR
    Device[设备] --> Buffer[单缓冲区]
    Buffer --> Process[用户进程]

3.3.4 双缓冲

原理

系统设置两个缓冲区，交替使用。

输入场景

  • 当进程处理缓冲区 1 的数据时
  • 设备可以同时往缓冲区 2 输入新数据

优势

  • 实现更好的重叠
  • 提高设备和 CPU 并行性
flowchart LR
    Device[设备] --> B1[缓冲区1]
    Device --> B2[缓冲区2]
    B1 --> Process[用户进程]
    B2 --> Process

3.3.5 循环缓冲

原理

分配一组缓冲区，首尾相连形成环。

适合场景

  • 连续数据流
  • 生产者消费者速度不匹配明显
  • 高吞吐持续 I/O

优点

  • 缓冲能力更强
  • 可持续吸收设备数据波动
flowchart LR
    B1[Buffer1] --> B2[Buffer2]
    B2 --> B3[Buffer3]
    B3 --> B4[Buffer4]
    B4 --> B1

3.3.6 三种缓冲对比

类型 缓冲数 并行性 实现复杂度 适用场景
单缓冲 1 一般 低 简单 I/O
双缓冲 2 较好 中 常见中高速 I/O
循环缓冲 多个 最好 较高 连续流式数据

3.3.7 缓冲为什么能减少中断次数

如果设备每来一个字节就中断一次，CPU 会被中断淹没。
有了缓冲后，可以：

  • 攒够一块数据再中断
  • 或 DMA 把一块搬完再通知 CPU

于是中断频率下降。

高频考点

  • 设置缓冲的目的
  • 单缓冲、双缓冲、循环缓冲区别
  • 缓冲如何提高并行性

易错点

  1. 缓冲区不是 Cache

    • Cache 主要面向高速访问局部性
    • I/O 缓冲主要面向设备与 CPU/内存速度协调

  2. 双缓冲不一定意味着速度翻倍

    • 只是改善重叠程度，不是机械加倍

  3. 缓冲不总是减少总数据量

    • 它优化的是传输组织方式

简答题模板

题目：为什么要设置 I/O 缓冲区？

答：
设置 I/O 缓冲区的主要目的是缓和 CPU 与 I/O 设备之间的速度不匹配，提高系统并行性和吞吐量。缓冲区能够临时存放输入或输出数据，使设备和 CPU 可以相对独立地工作；同时还能协调逻辑记录与物理记录大小不一致的问题，减少中断次数，放宽 CPU 对中断响应时间的要求，从而提高系统效率。

考前速背版

  • 缓冲作用：调速、协调大小、并行、减中断
  • 单缓冲：一个中转站
  • 双缓冲：两个交替，更易并行
  • 循环缓冲：多个成环，适合持续流

四、4.3 独占型外围设备的分配

4.1 设备独立性

一句话理解

设备独立性就是程序只说明“要什么设备”，而不说明“非哪台设备不可”。

4.1.1 问题提出

如果程序运行前指定了某台具体物理设备，分配确实简单，但有明显缺点：

  • 一旦这台设备故障，程序无法运行
  • 即使系统中有同类设备，也不能自动替换
  • 程序与设备绑定过紧，不灵活

4.1.2 什么是设备独立性

定义

用户通常不指定物理设备，而是指定逻辑设备。
操作系统负责建立：

  • 逻辑设备名
  • 物理设备名

之间的映射关系。

本质

把“使用需求”和“具体硬件”分离。

4.1.3 设备独立性的实现

操作系统需要提供：

  • 逻辑设备名与物理设备名的对应表
  • 设备分配策略
  • 逻辑到物理的转换机制
flowchart LR
    User[用户进程: 请求逻辑设备] --> Map[逻辑-物理映射表]
    Map --> Dev1[物理设备1]
    Map --> Dev2[物理设备2]
    Map --> Dev3[物理设备3]

4.1.4 设备独立性的优点

  1. 程序与物理设备无关

    • 更换设备不必改程序

  2. 提高可靠性

    • 一台坏了可切换同类设备

  3. 提高分配灵活性

    • OS 可以按当前资源状况分配最合适设备

  4. 有利于多道程序设计

    • 提高资源利用率

高频考点

  • 设备独立性的定义
  • 设备独立性的实现方式
  • 设备独立性的优点

易错点

  • 设备独立性不是“完全不需要物理设备”
  • 它只是让用户看不到物理设备细节

简答题模板

题目：什么是设备独立性？有什么优点？

答：
设备独立性是指用户程序通常不直接指定具体物理设备，而是使用逻辑设备名，由操作系统将逻辑设备名转换为实际物理设备名，从而实现程序与物理设备的分离。
其优点主要有：程序与具体设备无关，系统增减或更换设备时不必修改程序；当某台设备故障时，可由同类设备替代，提高系统可靠性；同时增加了设备分配的灵活性，有利于更有效地利用设备资源并支持多道程序设计。

考前速背版

  • 设备独立性：逻辑设备名替代物理设备名
  • 实现：逻辑名—物理名映射表
  • 优点：灵活、可靠、可移植

4.2 独占型外设分配

一句话理解

独占型设备一次只能给一个进程使用，所以分配策略很关键。

4.2.1 什么是独占型外围设备

独占型设备是指：

一次只能由一个进程独占使用的设备

例如：

  • 某台打印机
  • 某台磁带机

这类设备需要互斥管理。

4.2.2 两种分配方式

（1）静态分配

定义

进程在运行前就申请所需全部设备，满足后才允许运行。

优点

  • 实现简单
  • 能防止死锁

缺点

  • 设备利用率低
  • 某些设备在进程运行很长时间后才用到，却提前占住

（2）动态分配

定义

进程在运行过程中，什么时候需要设备就什么时候申请。

优点

  • 提高设备利用率
  • 更灵活

缺点

  • 可能产生死锁
  • 需要互斥同步控制（如信号量 PV 操作）

4.2.3 动态分配为何可能死锁

课件举例很典型：

  • 系统有 1 台打印机、1 台磁带机
  • 进程 P：先申请打印机，再申请磁带机
  • 进程 Q：先申请磁带机，再申请打印机

若：

  • P 先拿到打印机
  • Q 再拿到磁带机
  • 接着 P 等磁带机，Q 等打印机

就形成循环等待，发生死锁。

死锁形成图

graph LR
    P[进程P] -->|请求| T[磁带机]
    T -->|已分配给| Q[进程Q]
    Q -->|请求| PR[打印机]
    PR -->|已分配给| P

4.2.4 如何理解静态分配防死锁

静态分配要求：

  • 运行前一次性拿齐全部设备
  • 拿不齐就不运行

这样不会出现：

  • 先占有部分资源
  • 再等待其他资源

所以破坏了死锁必要条件中的“占有并等待”。

4.2.5 设备分配数据结构

课件提到两类重要结构：

1）设备类表

每类设备对应一栏，记录：

  • 设备类
  • 总台数
  • 空闲台数
  • 设备表起始地址等

作用：

  • 便于按“类”管理设备
  • 支持设备独立性

2）设备表

每台具体物理设备一项，记录：

  • 物理设备名
  • 逻辑设备名
  • 占有该设备的进程
  • 分配标志
  • 好/坏标志等

分配思路流程图

flowchart TD
    A[进程提出逻辑设备请求] --> B[查设备类表]
    B --> C{是否有空闲设备}
    C -- 否 --> D[阻塞/等待]
    C -- 是 --> E[查设备表并选定物理设备]
    E --> F[修改分配标志]
    F --> G[建立逻辑名到物理名映射]
    G --> H[分配成功]

高频考点

  • 独占型设备定义
  • 静态分配与动态分配比较
  • 动态分配为什么会死锁
  • 设备类表、设备表作用

易错点

  1. 静态分配不是效率最高

    • 它安全，但利用率低

  2. 动态分配不是一定死锁

    • 只是“可能产生死锁”

  3. 逻辑设备名不是设备类名

    • 逻辑设备名是对用户可见的抽象标识

简答题模板

题目：比较独占型设备的静态分配和动态分配

答：
独占型设备的静态分配是指进程在运行前一次性申请所需全部设备，只有满足后才允许运行。这种方式实现简单，并且能够防止死锁，但设备利用率较低。
动态分配是指进程在运行过程中按需要申请和释放设备，能够提高设备利用率和系统灵活性，但如果多个进程以不同顺序申请多个独占设备，就可能形成循环等待，从而导致死锁。
因此，静态分配偏安全，动态分配偏高效。

考前速背版

  • 独占设备：一次仅一个进程使用
  • 静态分配：运行前申请全部，防死锁、低利用率
  • 动态分配：随用随申请，高利用率、可能死锁
  • 结构：设备类表 + 设备表

五、4.4 共享型外围设备的驱动——磁盘

5.1 磁盘的物理结构

一句话理解

磁盘访问的本质就是：把磁头移动到正确磁道，等目标扇区转过来，再开始读写。

5.1.1 为什么磁盘重要

磁盘是典型的：

  • 块设备
  • 共享型设备
  • 操作系统重点优化对象

因为磁盘性能直接影响：

  • 文件系统速度
  • 程序装入速度
  • 虚拟内存换页速度
  • 数据库与服务器性能

5.1.2 磁盘基本结构

磁盘通常由多个盘片组成，每个盘片有两个盘面。每个盘面上有很多同心圆磁道。

基本概念

概念 解释
盘片 磁盘中的圆盘
盘面 盘片的一个表面
磁道 盘面上的同心圆
扇区 磁道划分出的固定小段
簇 相邻扇区组合成的分配单位
磁头 负责读写数据
柱面 不同盘面上相同半径位置的磁道集合

结构关系图

flowchart TD
    Disk[磁盘] --> Platter[多个盘片]
    Platter --> Surface[每个盘片两个盘面]
    Surface --> Track[每个盘面多个磁道]
    Track --> Sector[每个磁道多个扇区]
    Sector --> Cluster[相邻扇区组成簇]

5.1.3 柱面是什么

这是考试很喜欢考的概念。

定义

多个盘面上，半径相同的位置上的磁道，共同组成一个柱面。

为什么重要

如果要访问同一柱面上的不同盘面数据：

  • 通常不需要再移动磁头臂
  • 只需切换磁头即可

因此柱面是磁盘调度中的关键概念。

5.1.4 磁盘物理块地址

课件给出两种表述，本质一样：

  • (柱面号, 磁头号, 扇区号)
  • 有时也写成“面号/道号/扇区号”

注意：

  • “面”对应磁头选择
  • 柱面不是单独某一个面

5.1.5 磁盘读写方式

磁盘读写通常分三步：

  1. 移臂（寻道）
  2. 旋转等待
  3. 数据传送
flowchart LR
    A[移臂: 到目标柱面] --> B[旋转: 等目标扇区]
    B --> C[选磁头并读写]

典型例子

假设要读某个数据块：

  • 先把磁头移动到目标柱面
  • 再等该扇区转到磁头下方
  • 最后执行读写传输

因此磁盘 I/O 慢主要不是慢在“传输”，而是慢在：

  • 寻道
  • 旋转延迟

高频考点

  • 磁道、扇区、柱面定义
  • 磁盘物理地址表示
  • 磁盘读写三阶段

易错点

  1. 柱面不是一个盘面
  2. 簇不是物理最小单位
    • 扇区通常是物理读写基本单位
    • 簇更常是文件系统分配单位
  3. 磁盘慢不主要慢在数据传输本身
    • 主要慢在机械运动

简答题模板

题目：简述磁盘的物理结构

答：
磁盘通常由多个盘片组成，每个盘片一般有两个盘面。每个盘面上分布着若干同心圆磁道，每个磁道又划分为固定数量的扇区，相邻扇区可以组成簇。不同盘面上位置相同的磁道构成柱面。磁盘访问某个数据块时，通常需要先将磁头臂移动到目标柱面，再等待目标扇区旋转到磁头下方，最后进行数据传输。因此磁盘访问与其物理结构密切相关。

考前速背版

  • 盘片 → 盘面 → 磁道 → 扇区 → 簇
  • 相同半径磁道 = 柱面
  • 读写三步：寻道 → 旋转 → 传输

5.2 磁盘存取时间

一句话理解

磁盘存取时间 = 寻道时间 + 旋转延迟 + 数据传送时间。

5.2.1 磁盘存取时间公式

课件给出：

  • 存取时间：( T_a )
  • 寻道时间：( T_s )
  • 旋转速度：( r )（转/秒）
  • 传送字节数：( b )
  • 一个磁道中的字节数：( N )

标准表达

[
T_a = T_s + \frac{1}{2r} + \frac{b}{rN}
]

其中：

  • (\frac{1}{2r})：平均旋转延迟
    因为平均要等半圈
  • (\frac{b}{rN})：传送时间
    因为每秒可转 (r) 圈，每圈可传 (N) 字节

5.2.2 各部分含义

1）寻道时间 (T_s)

磁头从当前位置移动到目标磁道所需时间。

2）旋转延迟

目标扇区还没转到磁头下方，需要等待。

平均等待半圈，所以是：

[
\frac{1}{2r}
]

3）传送时间

磁头定位好后，实际把数据读/写出来的时间。

[
\frac{b}{rN}
]

5.2.3 为什么平均旋转延迟是半圈

因为目标扇区可能出现在任意位置：

  • 最好：刚好到达，等待 0
  • 最坏：刚错过，等 1 圈
  • 平均：等 1/2 圈

5.2.4 公式怎么考

常见考法：

  1. 已知转速、字节数、磁道容量，求传送时间
  2. 已知平均寻道时间，求总存取时间
  3. 比较不同调度策略是否主要优化了 (T_s) 或旋转延迟

典型分析

磁盘性能差异中，最重要影响项通常是：

  • 寻道时间

因为磁头臂机械移动开销通常比传输更大。

这也是为什么磁盘调度重点优化“移臂”。

高频考点

  • 存取时间公式
  • 平均旋转延迟为什么是半圈
  • 寻道时间的重要性

易错点

  1. r 的单位是转/秒，不是转/分

    • 若题目给 rpm，要先换算

  2. 平均旋转延迟不是 (1/r)

    • 是半圈，所以是 (1/(2r))

  3. 总存取时间不是只有寻道时间

    • 还包含旋转与传送

简答题模板

题目：写出磁盘存取时间公式并解释各项含义

答：
磁盘存取时间通常表示为：
[
T_a = T_s + \frac{1}{2r} + \frac{b}{rN}
]
其中，(T_a) 为总存取时间，(T_s) 为寻道时间，(\frac{1}{2r}) 为平均旋转延迟，表示平均等待半圈所需时间，(r) 为磁盘转速（转/秒），(\frac{b}{rN}) 为数据传送时间，其中 (b) 是要传送的字节数，(N) 是一个磁道包含的字节数。
该公式表明，磁盘访问时间由机械定位和实际传输两部分共同组成。

考前速背版

[
T_a = T_s + \frac{1}{2r} + \frac{b}{rN}
]

  • (T_s)：寻道
  • (1/2r)：平均半圈等待
  • (b/rN)：传送时间

5.3 磁盘驱动调度：移臂调度

一句话理解

移臂调度就是安排请求次序，让磁头少来回跑。

5.3.1 为什么需要磁盘调度

磁盘往往会同时收到多个 I/O 请求。
如果随机处理，会导致：

  • 寻道距离大
  • 总时间长
  • 吞吐量差

所以 OS 要决定：

先处理哪个请求，后处理哪个请求

5.3.2 移臂调度的目标

目标是：

  • 让磁头总移动距离尽量小
  • 从而减少寻道总时间

5.3.3 常见算法总览

算法 核心思想 优点 缺点
FCFS 按到达顺序 公平、简单 性能差
SSTF 选最近请求 寻道短、性能好 可能饥饿
SCAN 电梯算法，来回扫描 性能稳定 两端响应不均衡
LOOK 只扫到本方向最后请求 比 SCAN 少无效移动 仍有方向偏差
C-SCAN 单向循环扫描 等待时间更均匀 回绕有额外开销
C-LOOK 单向只扫到最后请求后跳回 更高效 实现略复杂
N-step-SCAN 分批扫描 避免粘性 实现复杂
FSCAN 双队列扫描 避免新请求干扰当前扫描 结构更复杂

5.3.4 FCFS（先来先服务）

思想

按请求到达顺序处理。

优点

  • 最公平
  • 最简单
  • 不会饥饿

缺点

  • 可能频繁大幅移臂
  • 性能通常较差

5.3.5 SSTF（最短寻道时间优先）

课件也称：

  • 最短查找时间优先
  • 最小短距法

思想

每次选择离当前磁头位置最近的请求。

优点

  • 通常比 FCFS 好
  • 减少短期寻道距离

缺点

  • 远处请求可能长期得不到服务
  • 会发生饥饿

5.3.6 SCAN（扫描/电梯算法）

思想

磁头像电梯一样沿一个方向移动：

  • 路上遇到请求就处理
  • 到头后再反向扫描

课件中的表述

  • 双向扫描
  • 当前方向没有请求时或到最后一个柱面后再改变方向
  • 改进版称电梯调度

优点

  • 性能稳定
  • 比 SSTF 更公平
  • 不易出现极端饥饿

缺点

  • 两端柱面可能等待时间较长
  • 中间区域响应更频繁

5.3.7 LOOK

思想

与 SCAN 类似，但不必真的走到磁盘物理尽头。
只要当前方向上没有请求了，就立刻掉头。

优点

  • 避免无意义移动
  • 比 SCAN 更高效

5.3.8 C-SCAN（循环扫描）

思想

磁头只沿一个方向提供服务。
到一端后，直接快速返回另一端，再继续同方向扫描。

为什么这样设计

为了让等待时间更均匀：

  • 不会因为方向不同导致某些请求老被延迟

特点

  • 更适合大量均匀分布请求
  • 服务更“循环公平”

5.3.9 C-LOOK

思想

类似 C-SCAN，但不走到物理尽头。
只扫到当前方向上最远的请求，然后跳回最小请求处重新开始。

优点

  • 比 C-SCAN 少无效移动
  • 保持单向扫描公平性

5.3.10 N-step-SCAN 与 FSCAN

这是课件补充内容，选择题或简答可能考。

N-step-SCAN

把请求队列分成长度为 N 的子队列，每次用 SCAN 处理一个子队列。

作用

防止某个进程持续往某一柱面发请求，形成“磁头粘性”。

FSCAN

用两个队列：

  • 一个队列正在扫描
  • 新到请求全部进入另一个队列

好处

当前扫描不受新请求干扰，更稳定。

5.3.11 算法关系图

flowchart TD
    A[磁盘调度]
    A --> B[FCFS]
    A --> C[SSTF]
    A --> D[SCAN]
    D --> E[LOOK]
    A --> F[C-SCAN]
    F --> G[C-LOOK]
    A --> H[N-step-SCAN]
    A --> I[FSCAN]

5.3.12 经典例题分析框架

课件给出经典序列：

  • 请求队列：98, 183, 37, 122, 14, 124, 65, 67
  • 磁头初始位置：53
  • 柱面范围：0 ~ 199

并给出结果：

  • FCFS：总移动 640
  • SSTF：总移动 236
  • SCAN：总移动 208

考试时解这类题怎么做

Step 1：标出当前磁头位置

初始在 53。

Step 2：按算法确定访问顺序

  • FCFS：按原顺序
  • SSTF：每次找最近
  • SCAN：先向某方向扫，再回来
  • C-SCAN：单方向扫并回绕

Step 3：求每一步移动距离

例如从 53 到 65，距离是 12。

Step 4：把所有移动距离加总

得到总移臂长度。

5.3.13 为什么 SSTF 不一定全局最优

因为 SSTF 只考虑当前这一步最短，是局部贪心。
局部最优不一定等于全局最优。

这点很适合简答题分析。

高频考点

  • FCFS、SSTF、SCAN、C-SCAN 比较
  • SSTF 为什么可能饥饿
  • SCAN 为什么叫电梯算法
  • LOOK 和 SCAN 区别
  • C-SCAN 和 SCAN 区别
  • 磁盘调度计算题

易错点

  1. SSTF 不保证平均寻道时间最小

    • 只是通常比 FCFS 更好

  2. SCAN 不一定走到最外柱面

    • 课本上不同定义有时区分 SCAN/LOOK，做题要看题目假设

  3. C-SCAN 返回时通常不提供服务

    • 只在一个方向服务请求

  4. 算总移动量时别漏掉“回到边界/跳回”的过程

    • 具体要看算法定义

简答题模板

题目：比较 FCFS、SSTF 和 SCAN 磁盘调度算法

答：
FCFS 按请求到达顺序处理，优点是实现简单、比较公平，不会产生饥饿现象，但磁头移动距离可能较大，因此性能较差。
SSTF 每次选择距离当前磁头位置最近的请求，能够有效减少短期寻道时间，通常性能优于 FCFS，但远处请求可能长期得不到服务，从而产生饥饿现象。
SCAN 又称电梯调度算法，磁头沿一个方向扫描，在移动过程中依次处理请求，到达该方向末端或无请求后再反向扫描。它兼顾了效率与公平性，性能较稳定，是一种常用磁盘调度算法。

考前速背版

  • FCFS：公平简单，性能差
  • SSTF：近的先做，效率高，会饥饿
  • SCAN：电梯来回扫
  • LOOK：只扫到最后请求就回头
  • C-SCAN：只单向服务，更均匀
  • FSCAN：双队列，防新请求扰动

5.4 磁盘驱动调度：旋转调度

一句话理解

移臂解决“到哪条道”，旋转调度解决“等哪个扇区先转到面前”。

5.4.1 旋转调度的目标

目标是：

  • 减少总旋转延迟
  • 尤其对同一柱面上的多个访问请求进行排序优化

5.4.2 循环排序

课件提到：

在最少旋转圈数内完成位于同一柱面的访问请求

这就是旋转调度的核心。

思想

如果多个目标块都在同一柱面上：

  • 移臂已不再是主要问题
  • 接下来关键是扇区旋转顺序

应该按扇区即将到达磁头的先后顺序访问，而不是按请求到达顺序死做。

5.4.3 课件中的经典例子

条件

  • 一个磁道有 4 个物理块
  • 转一周需 20ms
  • 有 4 个请求：读记录 4、3、2、1

三种处理方式

方法 1：按请求顺序 4→3→2→1

总处理时间 = 3 周 = 60ms

方法 2：按 1→2→3→4

总处理时间 = 1.5 周 = 30ms

方法 3：如果知道当前读位置是记录 3，则按 4→1→2→3

总处理时间 = 1 周 = 20ms

说明什么

同样的请求，只是换一下顺序，时间差别可能非常大。
这正是旋转调度存在的意义。

5.4.4 优化分布（交替地址）

除了调整请求顺序，还可以调整数据在磁道上的排列方式。

问题背景

如果逻辑上要顺序访问 A、B、C、D…
但处理完 A 后，CPU 来不及立刻接上 B，B 已经从磁头下转过去了，就得再等一圈。

解决方法

让逻辑相邻记录在物理上隔开一定间距，这叫：

  • 交替排序
  • 交替地址
  • 交叉因子（interleave factor）

5.4.5 交叉因子

若交叉因子为 n:1，表示相邻逻辑编号之间物理上间隔 n-1 个扇区。

目的

给 CPU 或控制器留出处理时间，使读取下一条逻辑记录时，它刚好转到位。

5.4.6 课件中的优化分布案例

原始顺序布局时，处理 10 条记录总时间：

  • 214ms

优化分布后，总时间变为：

  • 70ms

说明

合理的数据物理布局，可能比单纯调度还更有效。

旋转优化思想图

flowchart TD
    A[多个同柱面请求] --> B[按请求到达顺序处理]
    A --> C[按旋转顺序循环排序]
    C --> D[减少等待圈数]
    D --> E[缩短总旋转延迟]

    F[顺序逻辑记录] --> G[普通连续布局]
    F --> H[交替地址布局]
    H --> I[避免错过下一扇区]

  • 循环排序含义
  • 交替地址/交叉因子作用
  • 为什么优化布局能减少总时间

易错点

  1. 旋转调度主要针对同一柱面内请求
  2. 交替地址不是“随机打乱”
    • 是为了让逻辑顺序访问更快
  3. 旋转调度和移臂调度不是二选一
    • 两者配合使用

题目：什么是磁盘旋转调度？为什么要采用交替地址？

答：
磁盘旋转调度是指在磁头已经定位到目标柱面后，为了减少旋转延迟而对同一柱面上的多个访问请求进行排序优化，使其在尽可能少的旋转圈数内完成。
采用交替地址（交替排序）是因为在顺序读取多个逻辑记录时，处理完当前记录后，下一记录所在扇区可能已经从磁头下方转过，导致必须再等待一整圈。通过让逻辑相邻记录在物理上隔开一定间距，可以使下一条记录在系统准备好时恰好转到磁头下方，从而减少等待时间，提高磁盘访问效率。

考前速背版

  • 旋转调度：减少旋转等待
  • 循环排序：按最少转圈顺序服务
  • 交替地址：逻辑顺序 ≠ 物理紧挨，目的是更快顺序读

5.5 RAID 磁盘冗余阵列

一句话理解

RAID 是把多块磁盘组合起来，在性能、容量和可靠性之间做权衡。

5.5.1 RAID 的基本思想

RAID 全称：

  • 早期：Redundant Array of Inexpensive Disks
  • 后来：Redundant Array of Independent Disks

核心目标

  1. 提高 I/O 性能
  2. 增强容错能力
  3. 用多盘并行弥补单盘性能不足

5.5.2 RAID 的两大基础机制

1）条带化（Striping）

把连续数据分散到多块磁盘上并行读写，提高速度。

2）冗余（Redundancy）

通过镜像或校验保存额外信息，使部分磁盘损坏后仍能恢复数据。

5.5.3 常见 RAID 级别总览

RAID级别 主要机制 容错性 读性能 写性能 空间利用率 特点
RAID 0 纯条带化 无 高 高 高 只提速，不容错
RAID 1 镜像 高 较高 一般 50% 安全但成本高
RAID 2 海明码校验 有 一般 一般 较低 理论性强，少见
RAID 3 位/字节级条带 + 专用校验盘 可坏1盘 高 一般 较好 适合顺序访问
RAID 4 块级条带 + 专用校验盘 可坏1盘 较高 校验盘瓶颈 较好 小写入差
RAID 5 块级条带 + 分布式校验 可坏1盘 高 较好 较高 很常见
RAID 6 块级条带 + 双重校验 可坏2盘 高 略低 较高 安全性更高
RAID 10 镜像 + 条带 高 高 高 50% 兼顾性能与可靠性

5.5.4 各级别重点理解

RAID 0

  • 只有条带化
  • 没有冗余
  • 性能最好，风险最大
  • 任意一块盘坏，整体数据可能损坏

RAID 1

  • 镜像
  • 两份相同数据
  • 读性能好，可靠性高
  • 空间利用率低（约 50%）

RAID 5

  • 块级条带
  • 校验信息分布在各个磁盘上
  • 可坏 1 块盘
  • 是工程上很经典的一种平衡方案

RAID 6

  • 双重校验
  • 可同时坏 2 块盘
  • 比 RAID 5 更安全，但写入开销更大

RAID 10

  • 先镜像后条带
  • 兼顾性能和容错
  • 成本较高

RAID 关系图

flowchart TD
    RAID[RAID]
    RAID --> R0[RAID0: 条带化]
    RAID --> R1[RAID1: 镜像]
    RAID --> R5[RAID5: 分布式奇偶校验]
    RAID --> R6[RAID6: 双重校验]
    RAID --> R10[RAID10: 镜像+条带]

5.5.5 考试中怎么答 RAID

如果题目不要求特别细，重点答三点：

  1. RAID 用多盘并行提高性能
  2. RAID 用冗余提高可靠性
  3. 不同级别在性能、可靠性、成本之间权衡不同

高频考点

  • RAID 的定义与目的
  • RAID 0 / 1 / 5 / 6 / 10 区别
  • RAID 5 与 RAID 6 比较
  • 哪些 RAID 能容错

易错点

  1. RAID 0 不是冗余阵列中的“安全方案”

    • 它没有冗余

  2. RAID 不等于备份

    • RAID 提高可用性，不代替离线备份

  3. RAID 1 空间利用率低

    • 镜像需要双份空间

简答题模板

题目：简述 RAID 的思想及 RAID 5 的特点

答：
RAID 是将多块独立磁盘组合成一个逻辑存储系统，通过数据条带化实现并行读写，提高 I/O 性能；通过镜像或校验等冗余机制提高系统容错能力和可靠性。
RAID 5 采用块级条带化和分布式奇偶校验，数据块和校验块分散存放在各磁盘上。它能够在任意一块磁盘损坏的情况下继续工作，并在性能、可靠性和空间利用率之间取得较好的平衡，因此是应用较广的一种 RAID 级别。

考前速背版

  • RAID：多盘并行 + 冗余容错
  • RAID0：快，但不安全
  • RAID1：镜像，安全，费空间
  • RAID5：最常见，坏 1 盘还能用
  • RAID6：坏 2 盘还能用
  • RAID10：性能和安全都强，成本高

5.6 硬盘相关补充

一句话理解

5.6.1 磁盘 Cache

课件提到磁盘高速缓存，可理解为：

  • 利用局部性提高读写效率

作用

  • 提高命中时访问速度
  • 减少机械寻道和旋转开销
  • 常用于文件系统缓冲与页缓存

5.6.2 顺序访问和局部性

磁盘非常怕“乱跳”访问，喜欢：

  • 顺序访问
  • 成批访问
  • 局部集中的访问

所以操作系统会尽量：

  • 合并请求
  • 预读数据
  • 按柱面集中存储

高频考点

  • 为什么顺序访问快于随机访问

易错点

  • 顺序读快：少寻道、少旋转等待

六、4.5 虚拟设备

6.1 SPOOLing 系统

一句话理解

SPOOLing 用高速共享磁盘模拟慢速独占设备，让慢设备“看起来像随时可用”。

6.1.1 什么是虚拟设备技术

课件定义：

使用一类物理设备模拟另一类物理设备的技术

例子

  • 用磁盘模拟字符设备
  • 输入输出重定向
  • 用共享设备模拟独占设备

6.1.2 什么是 SPOOLing

SPOOLing 是：

  • Simultaneous Peripheral Operations On-Line
  • 中文常称：假脱机技术

本质

是一种：

  • 以共享型磁盘设备
  • 模拟独占型慢速字符设备

的技术。

为什么需要它

像打印机、读卡机这类设备：

  • 慢
  • 独占
  • 若进程直接等待它们，会严重拖慢系统

于是采用磁盘作为中介。

6.1.3 SPOOLing 的核心思想

输入方向

先把输入设备数据提前读到磁盘的输入井。

输出方向

先把进程结果写到磁盘的输出井，再由系统慢慢输出到打印机等设备。

于是，进程执行期间面对的主要 I/O 不再是慢字符设备，而是较快的磁盘 I/O。

6.1.4 SPOOLing 的软件组成

课件给出三部分：

  1. 预输入程序
  2. 缓输出程序
  3. 井管理程序

1）预输入程序

把数据从输入设备预先传送到磁盘输入井。

2）缓输出程序

把数据从磁盘输出井送往输出设备。

3）井管理程序

控制作业与输入井/输出井之间的数据交换。
本质上可理解为一种 I/O 重定向管理。

6.1.5 SPOOLing 系统结构图

flowchart LR
    InDev[输入设备] --> PreIn[预输入程序]
    PreIn --> InWell[输入井]
    InWell --> Job[运行作业]
    Job --> OutWell[输出井]
    OutWell --> PostOut[缓输出程序]
    PostOut --> OutDev[输出设备]
    Manager[井管理程序] --> InWell
    Manager --> OutWell

6.1.6 SPOOLing 的优点

  1. 把慢速独占设备变成“逻辑上的共享设备”
  2. 缩短进程驻留内存时间
  3. 提高系统吞吐量
  4. 提高设备利用率
  5. 实现主机与外设并行工作

6.1.7 为什么说 SPOOLing 是“速度匹配技术”

因为它让：

  • 慢速字符设备 ↔ 高速 CPU

之间的直接矛盾

转化为：

  • 慢速字符设备 ↔ 磁盘
  • CPU/进程 ↔ 磁盘

而磁盘比字符设备快得多，所以匹配更合理。

6.1.8 打印 SPOOLing 系统

课件重点例子：

打印过程

  1. 用户进程不直接占用打印机
  2. 先把完整待打印文件写入打印目录
  3. 打印机守护进程是唯一有权直接使用打印机的进程
  4. 打印机空闲时，守护进程从打印目录取任务并打印

这就是现实中的打印队列原理

你点了“打印”后：

  • 文档先进入打印队列
  • 你程序很快返回
  • 真正打印稍后后台完成

守护进程工作图

flowchart TD
    A[用户进程生成待打印文件] --> B[写入打印目录/输出井]
    B --> C[打印守护进程检测队列]
    C --> D{打印机空闲?}
    D -- 否 --> C
    D -- 是 --> E[取出队首任务]
    E --> F[占用打印机执行输出]
    F --> G[任务完成并释放打印机]

6.1.9 为什么 SPOOLing 能把独占设备“虚拟成共享设备”

注意，这里的“共享”是逻辑上的共享：

  • 物理上打印机仍然一次只能打印一个任务
  • 但多个进程可以同时“提交打印请求”
  • 这些请求先排队到磁盘中

因此从用户角度看，打印机像是“大家都能同时使用”。

高频考点

  • 虚拟设备定义
  • SPOOLing 的定义、本质和组成
  • 输入井/输出井作用
  • 打印假脱机原理
  • SPOOLing 为什么能提高吞吐量

易错点

  1. SPOOLing 不是让打印机真的并行打印多个任务

    • 只是通过排队和磁盘中介实现逻辑共享

  2. 输入井/输出井在磁盘上，不在外设上

  3. SPOOLing 是一种虚拟设备技术，也是一种速度匹配技术

简答题模板

题目：简述 SPOOLing 系统的原理及组成

答：
SPOOLing（Simultaneous Peripheral Operations On-Line）即假脱机技术，是利用共享型高速磁盘设备来模拟独占型慢速字符设备的一种虚拟设备技术，也是一种速度匹配技术。
其基本原理是：对输入设备，利用预输入程序先将数据输入到磁盘上的输入井；对输出设备，进程先将结果写入磁盘上的输出井，再由缓输出程序在设备空闲时输出到实际设备。井管理程序负责作业与输入井、输出井之间的数据交换。
通过这种方式，进程与慢速字符设备的直接耦合被打破，I/O 主要转化为较快的磁盘操作，从而提高系统吞吐量和设备利用率，并使独占设备在逻辑上表现为共享设备。

考前速背版

  • SPOOLing = 假脱机
  • 本质：共享磁盘模拟独占字符设备
  • 组成：预输入、缓输出、井管理
  • 井：输入井、输出井
  • 作用：逻辑共享、速度匹配、提高吞吐量

6.2 批处理系统的作业管理

一句话理解

在多道批处理系统中，SPOOLing 负责把作业“送进来/送出去”，作业调度负责决定“谁先进内存运行”。

6.2.1 作业状态

课件给出的作业状态包括：

  • 输入状态（预输入）
  • 后备状态
  • 运行状态
  • 完成状态（缓输出）

状态含义

状态 含义
输入状态 作业正在通过预输入进入系统
后备状态 已输入完成，等待调度进入运行
运行状态 已被选中并建立进程执行
完成状态 作业运行结束，结果等待缓输出

    B --> C[作业调度: 选中并创建进程]

6.2.3 作业调度与进程调度区别

这是很容易混淆的点。

比较项 作业调度 进程调度
又称 高级调度 低级调度
决定什么 哪些作业进入内存并建立进程 哪个就绪进程占用 CPU
作用层次 作业级 进程级
频率 较低 很高

关系

  • 作业先进入后备队列
  • 作业调度把作业选入内存，创建进程
  • 进程调度再决定这些进程谁先运行

  • 内存够不够
  • 设备资源够不够

简答题模板

题目：说明作业调度与进程调度的关系

答：
作业调度和进程调度分别对应操作系统中的高级调度和低级调度。作业调度从后备队列中选择合适作业进入内存，并为其创建进程；进程调度则从就绪队列中选择某个进程占用 CPU 执行。
因此，作业调度决定“哪些作业能进入运行环境”，进程调度决定“进入运行环境后的哪个进程先执行”。前者面向作业级管理，后者面向进程级管理，二者共同完成多道程序系统中的资源组织和运行控制。

  • 作业状态：输入 → 后备 → 运行 → 完成
  • 作业调度：决定谁进内存、谁建进程
  • 进程调度：决定谁上 CPU
  • SPOOLing：负责预输入和缓输出

八、期末简答题模板

模板 1：设备管理目标

答题模板：
设备管理的目标主要包括：

  1. 解决 CPU 与 I/O 设备速度不匹配的问题，使主机和外设尽可能并行工作，提高设备利用率和系统吞吐量；
  2. 屏蔽设备的物理细节和操作差异，为用户和应用程序提供统一接口；
  3. 实现设备独立性，使程序不依赖具体物理设备；
  4. 通过缓冲、调度、分配和虚拟设备等技术，提高系统效率与可靠性。

模板 2：I/O 控制方式比较

答题模板：
I/O 控制方式主要有轮询方式、中断方式和 DMA 方式。轮询方式中 CPU 发出 I/O 命令后不断查询设备状态，既等待设备又参与数据交换，效率最低；中断方式中 CPU 发出命令后可继续执行其他任务，设备完成后通过中断通知 CPU，因此 CPU 不等待设备，但仍参与后续处理；DMA 方式中由 DMA 控制器直接完成设备与主存之间的数据传输，CPU 只在开始和结束时参与，因此效率最高，适合大批量数据传输。

模板 3：I/O 软件层次

答题模板：
I/O 软件通常分为用户空间 I/O 软件、设备无关 I/O 软件、设备驱动程序、中断处理程序和 I/O 硬件。用户空间 I/O 软件向应用程序提供库函数和格式化 I/O；设备无关软件负责命名、保护、缓冲、分配和错误处理等通用功能；设备驱动程序把逻辑请求转化为具体物理操作；中断处理程序负责处理 I/O 完成或错误中断并唤醒相关进程；I/O 硬件最终完成实际的数据传输。

模板 4：缓冲区作用

答题模板：
设置 I/O 缓冲区的目的在于解决 CPU 与设备之间速度不匹配的问题，提高并行性和系统吞吐量；同时协调逻辑记录与物理记录大小不一致的问题，减少中断次数，并放宽 CPU 对中断响应时间的要求。常见缓冲技术有单缓冲、双缓冲和循环缓冲。

模板 5：设备独立性

答题模板：
设备独立性是指用户程序通常不直接指定物理设备，而是使用逻辑设备名，由操作系统将逻辑设备名映射到实际物理设备。它能使程序与具体设备解耦，便于设备更换和故障替代，提高系统可靠性和设备分配灵活性。

模板 6：磁盘调度算法比较

答题模板：
FCFS 按到达顺序处理请求，简单公平但性能较差；SSTF 每次选择距离当前磁头最近的请求，寻道性能较好，但可能导致远处请求饥饿；SCAN 采用电梯方式沿一个方向扫描并处理沿途请求，兼顾效率与公平性；C-SCAN 只沿一个方向提供服务，回绕后再继续扫描，等待时间更均匀。

模板 7：SPOOLing

答题模板：
SPOOLing 即假脱机技术，是利用共享型高速磁盘设备模拟独占型慢速字符设备的一种虚拟设备技术。它通过预输入程序将数据送入输入井，通过缓输出程序将结果从输出井送往输出设备，并由井管理程序控制数据交换。其作用是实现逻辑上的设备共享，缩短进程等待时间，提高系统吞吐量和设备利用率。

  • 高效

  • 可靠

  • DMA：CPU 不等 + DMA 搬

  • 设备无关
  • 驱动
  • 中断
  • 硬件

  • 并行
  • 减中断
  • 协调记录大小

设备分配

  • 动态：高利用率，可能死锁

磁盘结构

  • 盘片、盘面、磁道、扇区、柱面
  • 读写：寻道 → 旋转 → 传输
