一、整章总览：处理器管理到底在管什么

一句话理解

处理器管理就是：操作系统如何借助处理器硬件、中断机制、进程/线程抽象和调度算法，让 CPU 安全、高效、公平地为多个程序服务。

这一章的主线逻辑

可以把整章看成一个闭环：

flowchart LR
A[处理器硬件基础<br/>寄存器/指令/模式] --> B[中断机制<br/>让OS被激活]
B --> C[进程/线程管理<br/>描述执行实体]
C --> D[调度算法<br/>决定谁上CPU]
D --> E[进程切换/模式切换<br/>真正完成执行转换]
E --> B

你必须建立的整体认识

处理器管理不是零散知识点，而是一套完整机制：

  1. CPU 本身有寄存器、指令系统、工作模式。
  2. 操作系统平时并不是一直“主动运行”，而是依靠中断/异常/系统调用被激活。
  3. 为了管理“运行中的程序”，OS 引入了进程；为了更细粒度并发，又引入了线程。
  4. 当多个进程/线程都想运行时，需要调度算法决定先后次序。
  5. 调度做出决定后，要靠上下文切换与模式切换真正把 CPU 交给新实体。

二、2.1 处理器

2.1.1 处理器与寄存器

一句话理解

处理器是执行指令的核心硬件，而寄存器是处理器内部速度最快、最关键的状态存储单元。

1. 处理器的基本组成

课件给出了处理器的简单示意。核心包括：

  • 运算器（ALU）：完成算术、逻辑运算
  • 控制器：控制指令执行流程
  • 寄存器组：存储地址、数据、状态、指令等
  • 内部总线：连接各部件
  • 与主存、I/O 的接口部件

可视化理解

flowchart TB
CPU[CPU]
ALU[算术逻辑单元 ALU]
CU[控制单元]
REG[寄存器组]
MEM[主存]
IO[I/O设备]

CPU --> ALU
CPU --> CU
CPU --> REG
CPU <--> MEM
CPU <--> IO

2. 常见寄存器及作用

2.1 用户程序可见寄存器

这类寄存器程序员可以直接或间接使用，其目的主要是：

  • 减少访问主存的次数
  • 提高指令执行效率

包括：

  • 数据寄存器：又称通用寄存器，用于保存操作数、中间结果
  • 地址寄存器：保存地址信息，如：
    • 索引寄存器
    • 栈指针
    • 段地址寄存器等

2.2 控制与状态寄存器

主要用于控制处理器运行，通常由 OS 内核使用。

包括：

寄存器 含义 作用
PC Program Counter，程序计数器 存放下一条要取的指令地址
IR Instruction Register，指令寄存器 存放当前正在执行或刚取出的指令
CC Condition Code，条件码 反映运算结果，如正/负/零/溢出
Flag/状态位 中断允许、屏蔽、模式位等 反映和控制 CPU 状态

3. 程序状态字 PSW

定义

PSW（Program Status Word，程序状态字） 是记录当前程序运行动态信息的一组状态信息。

课件中强调两层含义：

  1. OS 概念上的 PSW：记录程序执行状态
  2. 硬件实现上的 PSW 寄存器：可能专设，也可能分散在一组控制/状态寄存器中

PSW 通常包含的内容

  • 程序计数器 PC
  • 指令寄存器 IR
  • 条件码 CC
  • 中断允许/禁止位
  • 中断屏蔽位
  • 处理器模式位
  • 内存保护位
  • 调试控制位等

为什么 PSW 很重要？

因为它本质上描述了“此刻 CPU 正处于什么状态”。

操作系统要实现：

  • 中断返回
  • 进程切换
  • 模式切换
  • 异常恢复

都离不开 PSW。

4. 核心概念辨析：寄存器、PSW、现场

概念 本质 关注点
寄存器 CPU 内部高速存储单元 存数据、地址、状态
PSW 状态信息集合/寄存器 描述程序执行状态
现场 程序被打断时 CPU 的当前状态 为了以后恢复执行

易错点

  • PSW 不是全部现场，但属于现场的重要组成部分
  • 现场不只是 PC，还包括通用寄存器、栈指针、PSW 等
  • PCB 不是现场本身，而是保存现场及管理信息的数据结构

5. 典型例子

假设程序正在执行加法：

CPU 可能会经历：

  1. 把 b、c 从主存取到寄存器
  2. ALU 完成加法
  3. 把结果写入寄存器或主存
  4. 根据结果设置条件码（如是否为 0）

此时：

  • 数据寄存器保存操作数/结果
  • PC 指向下一条指令
  • IR 保存当前指令
  • CC 反映运算结果状态

6. 高频考点

  • 用户可见寄存器与控制/状态寄存器区别
  • PC、IR、CC 的作用
  • PSW 的组成与作用
  • 为什么寄存器能提高执行效率

7. 易错点

  1. PC 不是当前指令内容，而是下一条指令地址
  2. IR 存的是指令本身，不是地址
  3. CC 是结果标志，不是调度优先级
  4. PSW 不是进程控制块 PCB
  5. 寄存器属于硬件，PCB 属于 OS 数据结构

8. 简答题模板

题目：简述 PSW 的作用

答题模板：

程序状态字 PSW 是记录当前程序运行动态信息的一组状态信息，通常包含程序计数器、条件码、中断允许/屏蔽位、处理器模式位等内容。PSW 一方面反映处理器当前执行状态，另一方面是中断处理、模式切换和进程切换时保存与恢复现场的重要依据，因此在操作系统中具有关键作用。

9. 考前速背版

  • 寄存器分两类：用户可见寄存器、控制与状态寄存器
  • PC：下一条指令地址
  • IR：当前指令
  • CC：运算结果标志
  • PSW：记录程序运行状态，是中断/切换恢复的重要依据

2.1.2 指令与处理器模式

一句话理解

指令决定 CPU 做什么，处理器模式决定“谁有权做什么”。

1. 机器指令是什么

机器指令是 CPU 能直接执行的基本命令，是处理器工作的最小功能单位之一。

一般包含：

  • 操作码字段：做什么操作
  • 操作数地址字段：操作谁
  • 可能还包含一些状态/特征信息

指令能完成什么

  • 算术运算
  • 逻辑运算
  • 数据传送
  • 控制流跳转

2. 指令执行过程

课件给出的标准过程是：

  1. 取指
  2. 解码
  3. 执行

Step 1：取指

CPU 根据 PC 从主存或 Cache 取出指令，放到 IR。

Step 2：解码

分析 IR 中的操作码，决定这条指令应该如何执行。

Step 3：执行

完成运算、数据传送或跳转；并更新结果和状态位。

Mermaid 流程图

flowchart LR
A[PC给出指令地址] --> B[从主存/Cache取指]
B --> C[指令送入IR]
C --> D[译码]
D --> E[执行微操作序列]
E --> F[写回结果/更新CC/修改PC]

3. 指令执行周期与流水线

指令执行周期

一条指令通常经历：

  • 取指
  • 解码
  • 执行

指令流水线

多个指令可以重叠执行不同阶段：

  • 第 1 条在执行
  • 第 2 条在解码
  • 第 3 条在取指

这样可以提高吞吐量。

指令\周期 1 2 3 4 5
指令1 取指 解码 执行
指令2 取指 解码 执行
指令3 取指 解码 执行

4. 特权指令与非特权指令

为什么要区分？

因为有些指令会直接影响系统核心资源，如果所有程序都能执行，系统会失控。

特权指令

只能由操作系统内核执行，例如：

  • 启动 I/O
  • 设置中断控制
  • 修改 PC/PSW 的某些关键位
  • 内存管理控制等

非特权指令

普通用户程序也可执行，如：

  • 加减乘除
  • 普通数据传送
  • 一般逻辑操作

对比表

类型 谁能执行 典型内容 风险
特权指令 内核态程序 启动I/O、改模式位、关中断 影响系统安全
非特权指令 所有程序 普通计算、传送、比较 风险较小

5. 处理器模式

定义

处理器模式是 CPU 运行时的权限级别，用于控制不同程序可执行的指令范围和可访问的资源范围。

课件提到可有 0/1/2/3 等多级保护环，但现代 OS 常用的是：

  • 内核模式（Kernel Mode）
  • 用户模式（User Mode）

常见对应关系

模式 权限 说明
内核模式 高 可执行特权指令，访问核心资源
用户模式 低 只能执行非特权指令

6. 为什么必须有模式切换

如果用户程序想访问设备、文件、内存管理结构，不能直接做，必须请求 OS 代办。

于是就要发生：

  • 用户态 → 内核态
  • 内核态 → 用户态

触发用户态到内核态的原因

  • 系统调用
  • 异常
  • 中断

返回用户态的方式

  • 内核处理结束后执行中断返回指令，如 iret

7. 模式切换不是进程切换

这是非常高频的辨析题。

项目 模式切换 进程切换
本质 权限级别变化 执行实体变化
是否一定换进程 不一定 一定
是否一定进内核 是 是
开销 相对较小 较大
例子 系统调用后又回到原进程 时间片到后换另一个进程

核心结论

  • 模式切换可以不换进程
  • 进程切换一定包含模式切换

8. 典型例子

用户程序调用 read()：

  1. 用户态执行库函数
  2. 触发系统调用陷入内核
  3. CPU 切到内核态
  4. OS 执行特权操作访问设备/文件系统
  5. 完成后返回用户态

这里可能：

  • 只发生模式切换：若调用很快完成并返回原进程
  • 同时发生进程切换：若该进程因 I/O 阻塞，CPU 转给别的进程

9. 高频考点

  • 特权指令和非特权指令区别
  • 为什么要引入处理器模式
  • 用户态与内核态区别
  • 模式切换的触发条件
  • 模式切换与进程切换区别

10. 易错点

  1. 系统调用本质上会引起用户态到内核态转换
  2. 用户态不能执行特权指令
  3. 中断/异常/系统调用都可触发模式切换
  4. 模式切换不一定导致进程切换

11. 简答题模板

题目：简述特权指令与处理器模式的关系

为了保护系统核心资源，处理器把指令划分为特权指令和非特权指令。特权指令只能在内核模式下执行，普通用户程序在用户模式下只能执行非特权指令。处理器模式的设置使操作系统能够限制用户程序对关键资源的直接访问，从而保证系统的安全性和稳定性。

12. 考前速背版

  • 指令执行三步：取指、解码、执行
  • 指令分：特权指令、非特权指令
  • 模式分：内核态、用户态
  • 用户态不能执行特权指令
  • 中断/异常/系统调用可触发用户态到内核态
  • 进程切换一定含模式切换，反之不一定

三、2.2 中断管理

2.2.1 中断

一句话理解

中断就是 CPU 正在干一件事时，被更紧急的事情打断，先去处理，再决定是否回来继续。

1. 中断的概念

课件定义：

程序执行过程中，遇到急需处理事件时，暂时中止 CPU 上现行程序运行，转去执行相应事件处理程序，待处理完成后再返回原程序被中断处或调度其他程序执行的过程。

关键词拆解

  • 暂时中止
  • 转去处理紧急事件
  • 处理完成后恢复或重新调度

2. 为什么说操作系统是“中断驱动”的

课件强调：

中断是激活操作系统的唯一方式。

更准确地理解是：

  • 用户程序在用户态运行
  • OS 内核通常不会无缘无故主动介入
  • 必须通过中断 / 异常 / 系统调用进入内核

也就是说，OS 的运行常常是“被事件触发”的。

图示：中断驱动的本质

flowchart LR
U[用户程序执行] --> T{是否发生事件?}
T -- 否 --> U
T -- 是: 中断/异常/系统调用 --> K[进入OS内核处理]
K --> R{处理结果}
R -- 返回原程序 --> U
R -- 切换到其他程序 --> N[新程序运行]

3. 广义中断与狭义中断

广义中断

广义上包括：

  • 中断（外部事件）
  • 异常（指令执行出错）
  • 系统异常/陷入（系统调用）

狭义中断

狭义通常只指：

  • 来自处理器外部
  • 与当前正在执行的指令无关
  • 如 I/O 中断、时钟中断、外部设备信号中断

4. 中断、异常、系统调用的区别

类别 来源 是否与当前指令有关 典型例子
中断 处理器外部 无关 时钟中断、键盘中断、I/O完成
异常 当前指令执行 有关 除零、地址越界、非法指令
系统调用/陷入 程序主动请求 有关 请求I/O、创建设备、创建进程

更标准的补充说明

从体系结构角度看：

  • 异常（exception） 多是同步的
  • 中断（interrupt） 多是异步的
  • 系统调用 往往通过陷入指令主动触发，属于同步进入内核

5. 进程状态转换与中断的关系

中断不是孤立的，它和进程三态紧密相关。

stateDiagram-v2
    [*] --> 就绪态
    就绪态 --> 运行态: 被调度
    运行态 --> 等待态: 等待I/O/资源/系统调用阻塞
    等待态 --> 就绪态: 事件完成
    运行态 --> 就绪态: 时间片到/被抢占

核心理解

  • 中断/异常/系统调用会激活内核
  • 内核根据事件结果，决定：
    • 返回原进程
    • 让进程阻塞
    • 抢占当前进程
    • 调度其他进程

6. 高频考点

  • 中断概念
  • 为什么说 OS 是中断驱动
  • 中断、异常、系统调用区别
  • 中断在进程状态转换中的作用

7. 易错点

  1. 系统调用不是“出错”，而是程序主动请求 OS 服务
  2. 异常通常和当前指令有关，中断通常和当前指令无关
  3. 中断处理完不一定返回原进程，也可能调度其他进程

8. 简答题模板

题目：说明中断、异常和系统调用的区别

中断、异常和系统调用都能使处理器进入内核态，但三者来源不同。中断通常来自处理器外部，与当前正在执行的指令无关，如时钟中断和 I/O 中断；异常由当前指令执行引起，如除零、非法指令和地址越界；系统调用则是用户程序通过陷入指令主动请求操作系统服务所引起的内核介入。三者共同构成操作系统被激活的重要机制。

9. 考前速背版

  • 中断：外部事件、异步
  • 异常：执行指令出错、同步
  • 系统调用：程序主动请求 OS 服务
  • OS 是中断驱动的
  • 中断后可能返回原进程，也可能切换进程

2.2.2 中断源

一句话理解

中断源就是“谁发起了打断 CPU 的请求”。

1. 中断源分类总览

课件列出四类重要中断源：

  1. 处理器硬件故障中断事件
  2. 程序性中断事件
  3. 自愿性中断事件
  4. I/O 中断事件
  5. 外部中断事件

2. 处理器硬件故障中断事件

来源

  • 处理器故障
  • 内存故障
  • 总线故障等

处理原则

课件给出的典型原则：

  • 保护现场
  • 停止设备
  • 停止 CPU
  • 向操作员报告
  • 等待人工干预

理解

这类问题一般不是普通程序能解决的，而是系统级严重错误。

3. 程序性中断事件

来源

处理器执行机器指令时产生的问题。

典型情况

  • 除数为零
  • 溢出
  • 非法指令
  • 用户态执行特权指令
  • 地址越界
  • 非法存取
  • 虚拟地址异常（如缺页）

处理方式

类型 处理方式
简单算术异常 报告用户或异常处理
非法指令/非法访问 终止进程
虚拟地址异常 调整内存后重新执行指令

易考点

缺页异常为什么常常不终止进程？

因为缺页是“执行环境暂时不满足”，不是程序必然错误。OS 只需把页面调入，再让该指令重执行。

4. 自愿性中断事件

这就是系统调用。

特征

  • 程序主动发出
  • 目的是请求 OS 服务
  • 常通过陷入指令触发

典型请求

  • 请求 I/O
  • 请求设备
  • 创建进程
  • 申请资源

处理流程

课件给出的关键词：

  • 陷入 OS
  • 保护现场
  • 根据功能号查入口地址
  • 跳转到具体处理程序

5. I/O 中断事件

来源

外围设备报告 I/O 状态。

典型情况

  • I/O 完成
  • I/O 出错
  • I/O 异常

OS 的动作

若 I/O 完成，通常要：

  • 调整进程状态
  • 唤醒等待进程
  • 把其加入就绪队列

为什么 I/O 中断很重要？

因为没有它，CPU 无法及时知道外设“已经做完了”。

6. 外部中断事件

包括：

  • 时钟/间隔时钟中断
  • 设备报到/结束中断
  • 键盘/鼠标信号中断
  • 关机/重启中断

最重要的是时钟中断

因为它支撑了：

  • 计时
  • 时间片轮转
  • 抢占调度

没有时钟中断，就很难实现现代抢占式分时系统。

7. 高频考点

  • 中断源分类
  • 程序性中断 vs 自愿性中断
  • 缺页异常的处理特点
  • I/O 中断与唤醒进程的关系
  • 时钟中断在调度中的作用

8. 易错点

  1. 系统调用是自愿性中断，不是硬件故障
  2. I/O 完成中断常导致等待进程转就绪
  3. 虚拟地址异常通常是“修复后重试”，不是立即终止
  4. 时钟中断是实现抢占的重要基础

9. 简答题模板

题目：简述常见中断源及其处理特点

常见中断源包括硬件故障中断、程序性中断、自愿性中断、I/O 中断和外部中断。硬件故障中断通常需要保护现场、停止设备并等待人工干预；程序性中断由指令执行异常引起，严重者终止进程，缺页等可修复异常则调整后重执行；自愿性中断即系统调用，由程序主动请求操作系统服务；I/O 中断用于报告设备完成或出错，常伴随等待进程唤醒；外部中断如时钟中断则支持记时与抢占调度。

  • 中断源：硬件故障、程序异常、系统调用、I/O、外部事件
  • 缺页：修复后重执行
  • I/O 完成：唤醒等待进程
  • 时钟中断：时间片与抢占基础

2.2.3 中断系统

一句话理解

中断系统就是“发现中断、决定是否响应、转到处理程序、再恢复运行”的整套软硬件机制。

1. 中断系统的组成

课件指出：

  • 硬件子系统：负责中断响应
  • 软件子系统：负责中断处理

这体现了软硬件协同设计思想。

图示

flowchart LR
A[中断源] --> B[中断装置/中断控制器<br/>硬件]
B --> C[保存PSW/PC 切入内核<br/>硬件]
C --> D[中断处理程序 ISR<br/>软件]
D --> E[恢复现场/调度/返回]

2. 中断装置

定义

负责发现并响应中断/异常的硬件装置。

不同来源的中断，发现机制不同

类型 发现/响应主体
外部中断 中断控制器
内部异常 指令控制逻辑/陷阱机构
系统调用 陷入指令触发系统陷阱

3. 中断控制器

作用

  • 接收外设发来的 IRQ
  • 在中断寄存器中记录中断
  • 在合适时机让 CPU 响应
  • 查询中断向量表找到服务程序地址

中断向量表是什么

中断向量表（Interrupt Vector Table） 是存放各类中断服务程序入口地址的表。

当发生某种中断时，系统通过“中断号/向量号”找到对应处理程序。

4. 陷阱与系统陷阱

陷阱（Trap）

通常指执行指令过程中检测到异常，转入异常处理程序。

系统陷阱

执行陷入指令后，转入系统调用处理程序。

区分

概念 来源 作用
陷阱 异常检测 处理指令执行异常
系统陷阱 系统调用指令 激活 OS 服务程序

5. 中断响应过程

课件给出的响应过程要点非常重要，几乎可以直接出简答题。

Step 1：发现中断源，提出中断请求

Step 2：检测中断寄存器中记录的中断

Step 3：判断是否被屏蔽

Step 4：若有多个中断，按优先级选择

Step 5：中断当前程序执行

Step 6：保存当前程序 PSW/PC 到核心栈

Step 7：转向 OS 的中断处理程序

Mermaid 流程图

flowchart TD
A[中断源提出请求] --> B[中断装置检测请求]
B --> C{是否屏蔽?}
C -- 是 --> Z[暂不响应]
C -- 否 --> D{是否多个中断?}
D -- 是 --> E[按优先级选择]
D -- 否 --> F[直接响应]
E --> F
F --> G[中断当前程序]
G --> H[保存PSW/PC到核心栈]
H --> I[进入中断处理程序]

6. 中断处理程序的任务

课件总结：

  • 保护未被硬件保护的处理器状态
  • 分析 PSW 中断码识别中断源
  • 查中断向量表并执行相应处理子程序
  • 恢复正常操作

“恢复正常操作”有两种情况

情况一：返回原进程

如果该中断不影响当前进程继续执行，则处理完直接返回。

情况二：不返回原进程，而进行调度

如果该中断导致：

  • 当前进程阻塞
  • 时间片用完
  • 更高优先级进程到达
  • 当前进程终止

那么 OS 会调整队列并启动调度。

C --> D[交换/保存PSW]

F --> G[执行相应处理程序]
G --> H{是否需要调度?}

I --> J[中断返回]

O --> P[新进程运行]

  • 中断向量表的作用

  • 中断处理过程

9. 易错点

  1. 中断响应主要靠硬件，中断处理主要靠软件
  2. 保存 PC/PSW 往往由硬件先做一部分，但完整现场通常还需软件补充保存
  3. 中断处理结束不一定回原程序
  4. 中断向量表存的是入口地址，不是处理代码本身的全部内容

题目：简述中断响应与中断处理过程

中断响应是由硬件完成的，主要过程包括：发现中断源、提出中断请求、检测是否允许中断、按优先级选择中断、保存当前程序的 PSW/PC 并转入操作系统中断处理程序。中断处理是由软件完成的，主要包括保护未保存的现场、分析中断源、执行相应的中断服务程序、调整进程状态和队列，并根据情况恢复原进程或启动调度切换到新进程。

  • 处理后两种结果：回原进程 / 调度新进程

2.2.4 多中断的响应与处理

一句话理解

多个中断同时或嵌套出现时，要靠屏蔽、优先级和嵌套规则来维持系统有序。

1. 中断屏蔽

定义

通过中断屏蔽位决定某类中断是否允许被响应。

作用

  • 选择性响应中断
  • 防止关键处理过程被不合适地打断
  • 保证某些原子操作的完整性

2. 中断优先级

定义

多个中断同时到达时，决定先响应哪个。

课件示例中的可能次序

  • 处理器硬件故障
  • 自愿性中断
  • 程序性中断
  • 时钟等外部中断
  • I/O 中断
  • 重启动和关机中断

注意：不同系统实现可以不同，考试一般答“系统按预定优先级响应”。

3. 中断嵌套

定义

在处理中断 A 的过程中，又响应了中断 B。

为什么允许？

因为有的中断更紧急，不能等。

为什么要限制层数？

因为每嵌套一层，都要：

  • 继续保存现场
  • 增加系统开销
  • 增加控制复杂度

所以一般要限制嵌套层数。

4. 三者关系：屏蔽、优先级、嵌套

机制 解决什么问题
中断屏蔽 某个中断能不能响应
中断优先级 多个中断先响应谁
中断嵌套 响应后处理中能否再被打断

5. 典型情形分析

情形 1：X、Y 同时发生，Y 被屏蔽

  • 先响应 X
  • 因 Y 被屏蔽，不响应 Y
  • X 处理完后再看是否可处理 Y

情形 2：X、Y 同时发生，Y 未屏蔽且优先级更高

  • 可能先按规则响应 X
  • 处理中若允许嵌套，Y 可以打断 X
  • 先处理 Y，再回来处理 X

6. Mermaid 图示

flowchart TD
A[多个中断到达] --> B{是否被屏蔽?}
B -- 被屏蔽 --> C[暂不响应]
B -- 未屏蔽 --> D[按优先级选中断]
D --> E[进入处理]
E --> F{处理中是否允许更高优先级中断嵌套?}
F -- 否 --> G[处理完当前中断]
F -- 是 --> H[响应更高优先级中断]
H --> I[处理嵌套中断]
I --> G

  • 什么是中断屏蔽
  • 什么是中断优先级
  • 什么是中断嵌套
  • 三者区别与联系
  • 为什么中断嵌套层数要受限

8. 易错点

  1. 优先级高不代表一定立刻响应，前提是没被屏蔽
  2. 先响应的不一定先处理完，因为可能被更高优先级中断嵌套
  3. 中断嵌套提升实时性，但会增加开销

在多重中断环境下，中断屏蔽用于决定某类中断是否允许被响应；中断优先级用于在多个未屏蔽中断同时存在时确定响应顺序；中断嵌套则表示在处理中断过程中再次响应新的中断。三者共同保证系统能够有选择地、按重要程度处理多个中断，同时兼顾实时性与系统开销。

四、2.3 进程管理

2.3.1 进程及其状态

一句话理解

进程就是操作系统为了管理“正在运行的程序”而建立的执行实体。

1. 为什么要提出进程

课件的逻辑很直接：

  • OS 必须全方位管理运行中的程序
  • 所以要为每个运行中的程序建立一个管理实体
  • 这个实体就是进程

关键思想

程序只是静态代码文件，进程才是动态运行实体。

2. 进程的概念

课件定义：

进程是一个具有一定独立功能的程序关于某个数据集合的一次运行活动。

还强调：

进程是操作系统进行资源分配和处理器调度的一个独立单位。

两层含义都要会写

  1. 运行观点：程序的一次执行
  2. 管理观点：资源分配和调度的基本单位（经典说法）

3. 进程的形式化组成

课件给出五部分：

  • P：OS 管理运行程序的数据结构
  • C：内存代码
  • D：内存数据
  • R：通用寄存器信息
  • PSW：程序状态字信息

即：

一个进程 = 管理信息 + 程序代码 + 数据 + 寄存器上下文 + 状态信息

这很重要，因为它说明进程绝不是“只有程序代码”。

4. 程序与进程的区别

项目 程序 进程
性质 静态 动态
存在形式 文件/指令集合 运行活动
生命周期 长 有创建和撤销
是否占用资源 不一定 必然占用
是否可并发 程序本身不谈并发 进程可以并发

高频结论

  • 同一个程序可对应多个进程
  • 同一个外存文件，两次运行是两个不同进程
  • 不同程序也可形成不同进程
  • 相同代码可共享，但进程仍不同

5. 进程的典型例子

情形 1：不同程序，不同数据集

形成两个无关进程。

情形 2：不同程序，相同数据集

形成共享数据的交往进程。

情形 3：相同代码，不同数据集

形成两个共享代码的无关进程。

什么是可再入程序

课件提到：

  • 可再入程序可被多个进程共享
  • 它应是“纯代码”，不依赖被破坏的共享可修改状态

6. 进程的三态模型

三种基本状态

  • 运行态：占有处理器执行
  • 就绪态：具备运行条件，等待 CPU
  • 等待态（阻塞态）：因等待资源、I/O、信号等暂时不能运行

状态转换图

stateDiagram-v2
    [*] --> 就绪态
    就绪态 --> 运行态: 被选中
    运行态 --> 等待态: 等待资源/I-O/信号
    等待态 --> 就绪态: 事件完成
    运行态 --> 就绪态: 时间片到/被抢占

课件特别强调的两个“不可能”

  • 不可能从等待态直接到运行态
  • 不可能从就绪态直接到等待态

为什么？

因为：

  • 等待态结束后，先变成就绪态，再等待调度
  • 就绪态本来就在等 CPU，没有执行，就不可能主动发出阻塞请求

这是考试特别爱问的判断题。

7. 挂起状态

为什么要引入挂起

当系统资源不足，尤其是内存不足时，为了缓解负载：

  • 可把某些进程从内存换出
  • 放到外存对换区
  • 暂时不参与调度

这就是挂起

挂起 vs 等待 的本质区别

项目 等待态 挂起态
是否在内存中 一般在 可不在
是否保留已获资源 通常保留部分 常被剥夺主要资源
产生原因 等待事件 系统负载/内存不足
是否参与调度 否 否

挂起扩展状态

  • 挂起就绪态
  • 挂起等待态

    运行态 --> 就绪态: 时间片到/抢占

    就绪态 --> 挂起就绪态: 挂起

8. 进程状态模型补充理解

现代系统中状态可能更细：

  • 新建
  • 就绪
  • 运行
  • 阻塞
  • 挂起就绪
  • 挂起阻塞
  • 终止

考试时：

  • 若问“基本三态”，答三态
  • 若问“扩展状态模型”，答挂起/新建/终止

  1. 进程不是程序文件
  2. 进程是动态概念，程序是静态概念
  3. 就绪态并不缺资源，只缺 CPU
  4. 等待态是因事件未完成，缺的是运行条件
  5. 挂起不是普通阻塞，它涉及换出内存与资源剥夺

11. 简答题模板

题目：说明进程的概念及三态模型

进程是一个具有一定独立功能的程序关于某个数据集合的一次运行活动，也是操作系统进行资源分配和处理器调度的独立单位。进程的基本状态包括运行态、就绪态和等待态。运行态表示进程正在占用 CPU 执行；就绪态表示进程具备运行条件，只等待 CPU；等待态表示进程因等待 I/O、资源或事件而暂时不能执行。进程在这些状态之间通过调度、阻塞、唤醒和抢占实现转换。

12. 考前速背版

  • 进程 = 程序的一次运行活动
  • 进程 = 资源分配和处理器调度单位
  • 三态：运行、就绪、等待
  • 等待不能直达运行；就绪不能直达等待
  • 挂起是被换出，不参与调度

2.3.2 进程的数据描述

一句话理解

操作系统要管理进程，必须先“把进程描述清楚”，而 PCB 就是这张总档案。

1. PCB：进程控制块

定义

PCB（Process Control Block） 是操作系统用于记录和刻画进程状态及环境信息的数据结构。

为什么 PCB 是进程存在的标志？

因为操作系统并不是“凭感觉”管理进程，而是靠 PCB 管理：

  • 进程是谁
  • 现在什么状态
  • 在哪里
  • 占了什么资源
  • 下次怎么恢复执行

通常说：

PCB 是进程存在的唯一标志（经典表述）

2. PCB 的三大类信息

课件将 PCB 内容概括为：

  • 标识信息
  • 现场信息
  • 控制信息

2.1 标识信息

用于唯一标识进程，例如：

  • 进程 ID
  • 进程组 ID
  • 进程名
  • 进程组名

2.2 现场信息

用于保存进程被切换时的处理器现场：

  • 通用寄存器内容
  • 控制/状态寄存器内容（PC、IR、PSW）
  • 用户栈/核心栈指针

2.3 控制信息

用于调度与管理：

  • 进程状态
  • 等待原因
  • 优先级
  • 代码/数据地址
  • 外存映像地址
  • 队列指针
  • 父子兄弟进程指针
  • 通信信息
  • 特权信息
  • CPU 使用时间/时间片/记账信息
  • 资源清单

2.3.3 进程管理的实现

一句话理解

进程管理不是空谈概念，而是依靠队列、原语和控制程序真正做出来的。

1. 进程管理软件包括什么

课件列出关键组成：

  • 系统调用/中断/异常处理程序
  • 队列管理模块
  • 进程控制程序
  • 进程调度程序
  • 进程通信程序
  • 登录、作业控制、监控等外围程序

核心中的核心

其中最关键的是：

  • 队列管理
  • 进程控制原语
  • 调度程序

2. 队列模型

进程管理围绕各种队列展开：

  • 就绪队列
  • 各种等待队列（按事件分类）
  • 挂起相关队列

图示

flowchart LR
A[提交/创建] --> B[就绪队列]
B --> C[处理器运行]
C --> D{发生什么?}
D -- 时间片到/被抢占 --> B
D -- 等待事件1 --> E[事件1等待队列]
D -- 等待事件2 --> F[事件2等待队列]
D -- 结束 --> G[撤销]
E -- 事件1完成 --> B
F -- 事件2完成 --> B

为什么要多个等待队列？

因为不同进程可能等待不同事件：

  • 等待某设备完成
  • 等待某信号量
  • 等待某消息
  • 等待某资源

分队列便于唤醒和管理。

3. 队列管理模块

作用

  • 维护各种进程队列
  • 支持入队、出队
  • 根据策略组织为 FIFO 或优先队列
  • 通过 PCB 中的队列指针连接各 PCB

为什么说它是核心模块？

因为进程状态变化，本质上常表现为：

从一个队列移到另一个队列

例如：

  • 就绪 → 运行
  • 运行 → 等待
  • 等待 → 就绪

4. 进程控制原语

课件列出了典型操作：

  • 进程创建
  • 进程撤销
  • 进程阻塞
  • 进程唤醒
  • 进程挂起
  • 进程激活

各原语含义

进程创建

  • 进程表加项
  • 申请 PCB 并初始化
  • 分配标识
  • 建立映像
  • 分配资源
  • 入就绪队列

进程撤销

  • 从队列中移除
  • 回收资源
  • 回收 PCB
  • 删除进程表项

进程阻塞

  • 保存现场
  • 修改 PCB 状态
  • 入等待队列
  • 调度其他进程

进程唤醒

  • 从等待队列移出
  • 修改状态
  • 入就绪队列
  • 若优先级高，可触发抢占

进程挂起

  • 修改状态
  • 出入相关队列
  • 回收主存等资源
  • 送对换区

进程激活

  • 调回内存
  • 恢复资源
  • 修改状态
  • 回到合适队列

5. 什么是原语（Primitive）

定义

原语是由若干条指令组成、完成特定功能、执行上不可分割的程序段。

关键词

  • 不可分割
  • 原子性
  • 常通过关中断等方式保证

为什么进程控制必须用原语？

因为进程控制要修改核心数据结构：

  • 进程表
  • PCB 池
  • 队列
  • 资源表

如果执行到一半被打断，可能产生“与时间有关的错误”，即数据结构混乱。

6. 高频考点

  • 进程管理软件构成
  • 队列管理模块作用
  • 典型进程控制原语
  • 原语的定义与特点
  • 为什么进程控制必须是原子操作

7. 易错点

  1. 原语不是单条机器指令，而是一段不可分割的程序
  2. 原语的重点是执行原子性
  3. 阻塞后要进等待队列，不是直接删除
  4. 唤醒后先进就绪队列，不是直接运行

8. 简答题模板

题目：为什么进程控制操作必须由原语实现？

进程控制操作需要修改进程表、PCB、进程队列和资源表等操作系统核心数据结构。如果这些操作在执行过程中被中断，就可能造成数据不一致或系统状态混乱，产生与时间有关的错误。因此，进程控制操作必须通过具有原子性的原语实现。原语虽然由多条指令组成，但执行时不可分割，常通过关中断等方式保证其完整性。

9. 考前速背版

  • 进程管理核心：队列 + 原语 + 调度
  • 原语特点：不可分割、原子性
  • 阻塞：运行→等待
  • 唤醒：等待→就绪
  • 创建：建 PCB、建映像、分资源、入就绪

2.3.4 进程切换与模式切换

一句话理解

进程切换是“换人上 CPU”，模式切换是“换权限级别”。

1. 进程切换的定义

课件定义：

从正在运行的进程中收回处理器，让待运行进程占有处理器运行。

本质上：

被中断运行进程 与 待运行进程 的上下文切换

2. 进程切换的三大步骤

Step 1：保存被中断进程的上下文

Step 2：转向进程调度

Step 3：恢复待运行进程的上下文

这是最经典的答题框架。

3. 为什么进程切换必须在内核态完成

因为进程切换涉及：

  • 保存/恢复寄存器
  • 修改 PCB
  • 修改进程状态
  • 修改地址空间
  • 切换队列
  • 调用调度程序

这些都属于特权操作，必须在内核模式下进行。

4. 模式切换回顾

用户态 → 内核态

由以下事件触发：

  • 中断
  • 异常
  • 系统调用

内核态 → 用户态

由中断返回指令触发，把控制权交还用户进程。

5. 模式切换的具体工作

正向模式切换（用户 → 内核）

中断装置完成：

  • 模式位置内核态
  • 保存当前进程 PC/PSW 到核心栈
  • 跳转到中断/异常/系统调用处理程序

逆向模式切换（内核 → 用户）

中断返回指令完成：

  • 从待运行进程核心栈中弹出 PSW/PC
  • 恢复用户态
  • 继续执行

6. 进程切换的完整流程

课件给出的 12 步非常重要。

标准流程

  1. 中断/异常等触发正向模式切换并压入 PSW/PC
  2. 保存被中断进程的现场信息
  3. 处理具体中断/异常
  4. 把被中断进程系统栈指针 SP 保存到 PCB
  5. 调整被中断进程 PCB 信息（如状态）
  6. 把被中断进程 PCB 加入相关队列
  7. 选择下一个占用 CPU 的进程
  8. 修改被选中进程 PCB 信息（如状态）
  9. 设置被选中进程地址空间，恢复存储管理信息
  10. 恢复被选中进程 SP 到处理器寄存器
  11. 恢复被选中进程现场信息
  12. 中断返回，逆向模式切换并弹出 PSW/PC

7. 什么时候会发生进程切换

课件列出了三类典型情况：

1）阻塞式系统调用 / 缺页异常

当前进程进入等待态，需要换别人运行。

2）时间片中断 / 更高优先级进程到达

当前进程从运行态转就绪态，被抢占。

3）终止调用 / 无法继续执行的异常

当前进程进入终止态，必须换别的进程。

8. 什么情况不会引起进程切换

有些中断/异常只是短暂处理一下，处理后还回原进程：

  • 保存现场
  • 处理中断/异常
  • 恢复现场
  • 返回原进程

例如某些不影响继续执行的内核服务处理。

9. 模式切换与进程切换对比

项目 模式切换 进程切换
是否换执行主体 不一定 一定
是否切换地址空间 通常不一定 通常要
是否修改 PCB/队列 一般不必 必须
开销 较小 较大
关系 可单独发生 必然伴随模式切换

  • 为什么进程切换要在内核态
  • 模式切换步骤
  • 两者区别与联系
  • 哪些事件会/不会引起进程切换

11. 易错点

  1. 系统调用一定有模式切换，但不一定有进程切换
  2. 时间片到一般会导致抢占，从运行态到就绪态
  3. 恢复现场时恢复的是“将要运行的进程”的现场
  4. 进程切换比模式切换开销大得多

12. 简答题模板

题目：简述进程切换过程

进程切换是指操作系统收回当前运行进程的处理器使用权，并把处理器分派给另一个待运行进程。其本质是进程上下文的切换。一般过程包括：首先由中断、异常或系统调用触发进入内核态；然后保存当前进程的现场和系统栈指针，修改其 PCB 状态并放入相应队列；接着由调度程序选择新的进程，恢复其地址空间、栈指针和现场信息；最后通过中断返回指令恢复用户态，使新进程开始运行。

  • 模式切换：用户态↔内核态
  • 进程切换一定含模式切换
  • 系统调用不一定导致进程切换

五、2.4 多线程技术

2.4.1 多线程环境概述

一句话理解

线程把“资源拥有者”和“执行者”分离，让并发更轻、更快、更细。

1. 单线程进程的问题

传统进程是单线程结构：

  • 一个进程只有一条执行流
  • 一个 PCB
  • 一个用户地址空间
  • 一个执行序列

存在的问题

课件列出：

  • 进程切换开销大
  • 进程通信开销大
  • 限制并发粒度（进程内不能并发）
  • 降低并行计算效率（只有一个执行流）

2. 为什么引入线程

核心思路：

把进程的两项功能分离：

  1. 资源分配与保护
  2. 被调度分派执行

于是：

  • 进程负责资源拥有与保护
  • 线程负责执行与调度

这就是线程思想的本质。

3. 多线程结构进程

多线程进程中：

  • 多个线程共享同一进程地址空间和资源
  • 每个线程有自己的执行序列
  • 每个线程有自己的栈、寄存器上下文、线程控制块

图示

AS[共享地址空间]
R[共享资源]
T1[线程1: PC/寄存器/栈]
T2[线程2: PC/寄存器/栈]
T3[线程3: PC/寄存器/栈]

P --> AS

4. 多线程环境下的进程概念

在多线程环境中，进程是：

  • 资源分配和保护的独立单位
  • 拥有虚拟地址空间
  • 拥有文件/设备访问保护机制

换句话说：

进程更像“容器”

5. 多线程环境下的线程概念

线程是：

  • 进程中的一条执行路径
  • 调度的基本单位
  • 共享所属进程的主存空间和资源

线程具有：

  • 执行状态
  • 线程上下文
  • 独立程序计数器
  • 执行栈
  • 局部变量存储

线程与进程对比

项目 进程 线程
资源拥有者 是 否（共享所属进程资源）
调度单位 传统系统中常是 现代系统常是
地址空间 独立 共享进程地址空间
切换开销 大 小
通信开销 大 小

6. 线程状态

课件给出线程状态：

  • 运行
  • 就绪
  • 睡眠/阻塞

并强调：

  • 挂起状态与资源相关，属于进程，不属于线程

这点很重要。

7. 线程调度的两种环境

OS 感知线程

  • 处理器调度对象是线程
  • 进程不再体现经典三态（或只有挂起层面的意义）

OS 不感知线程

  • OS 调度对象还是进程
  • 线程由用户空间线程库调度

这恰好对应后面的 KLT 与 ULT。

8. 多线程的优点

课件列出：

  • 快速线程切换
  • 减少系统管理开销
  • 线程通信易实现
  • 提高并行程度
  • 节省内存空间

为什么通信更容易？

因为线程共享进程地址空间，直接访问共享变量即可；而进程间通信通常要借助 OS 机制。

9. 多线程的应用场景

  • 前台/后台工作
  • C/S 应用
  • 提升执行速度
  • 设计用户界面响应

例如浏览器：

  • 一个线程负责 UI
  • 一个线程负责网络
  • 一个线程负责渲染
  • 一个线程负责脚本执行

10. 高频考点

  • 为什么引入线程
  • 进程与线程关系
  • 线程的属性
  • 多线程的优点
  • 为什么挂起属于进程而不是线程

11. 易错点

  1. 线程是调度单位，进程是资源分配单位
  2. 同一进程中的线程共享地址空间，但不共享寄存器和栈
  3. 线程切换通常比进程切换轻量
  4. 挂起主要和进程资源、内存换出有关

12. 简答题模板

题目：为什么多线程比单线程进程更适合并发？

在单线程进程中，一个进程只有一条执行流，进程切换和进程间通信开销较大，且进程内部无法并发，限制了并发粒度。多线程技术把资源分配和执行调度分离开来，进程作为资源拥有者，线程作为执行和调度的基本单位。多个线程共享同一进程的地址空间和资源，因此线程切换开销较小、线程间通信方便，并且能够提高系统并发性和并行计算效率。

13. 考前速背版

  • 进程管资源，线程管执行
  • 线程是调度基本单位
  • 同进程线程共享地址空间和资源
  • 线程有自己的 PC、寄存器、栈
  • 线程切换比进程切换轻量

2.4.2 KLT 与 ULT

一句话理解

KLT 由内核管、能真并行；ULT 由用户管、切换更轻但易整进程阻塞。

1. KLT：内核级线程

定义

线程管理工作由 OS 内核完成，OS 直接感知并调度线程。

特点

  • 一个线程阻塞，内核可调度同进程其他线程
  • 多处理器下可让同进程多个线程并行
  • 调度与管理在内核中实现
  • 线程切换往往需要模式切换，开销较大

优势

  • 支持真正并行
  • 阻塞影响局部，不拖垮整个进程
  • 内核统一管理，功能强

缺点

  • 线程切换成本较高
  • 内核参与多，系统开销较大

2. ULT：用户级线程

定义

线程库在用户空间实现线程管理，内核不知道线程存在。

特点

  • 所有线程数据结构都在用户空间
  • 线程切换不必进入内核
  • 可按应用需求自定义调度算法
  • 可运行在不支持线程的 OS 上

缺点

  • 一个线程阻塞，整个进程阻塞
  • 不能充分利用多处理器真正并行
  • 内核看到的仍只是一个进程

3. KLT 与 ULT 对比表

对比项 KLT ULT
管理者 OS 内核 用户空间线程库
OS 是否感知线程 感知 不感知
调度对象 线程 OS 调度进程，线程由库调度
切换是否需内核介入 通常需要 不需要
切换开销 较大 较小
阻塞影响 单线程阻塞不影响其他线程 一个线程阻塞整个进程阻塞
多核并行 可以 一般不行
调度灵活性 内核统一 应用自定义强

4. Jacketing 技术

作用

将阻塞式系统调用改造成尽量不把整个 ULT 进程拖死的方式。

基本思路

当线程陷入系统调用时：

  • 由 jacketing 程序先检查资源情况
  • 决定是否做进程切换或控制权转给另一个线程

理解

它是 ULT 为弥补“一个线程阻塞整个进程阻塞”的缺陷而采取的折中技巧。

5. 课件中的经典结论

  • ULT 适合解决逻辑并行性问题
  • KLT 适合解决物理并行性问题

如何理解

  • 逻辑并行：程序结构上看起来多个任务并发
  • 物理并行：在多核 CPU 上真正同时跑多个线程

6. 高频考点

  • KLT 与 ULT 的定义
  • 两者优缺点
  • 为什么 ULT 切换快
  • 为什么 ULT 不能很好利用多处理器
  • 为什么 KLT 更适合并行计算
  • Jacketing 技术作用

7. 易错点

  1. ULT 不是没有调度，而是由用户空间调度
  2. KLT 切换一般更重，因为需要内核介入
  3. ULT 的阻塞问题是整进程级别的
  4. 多核上真正并行通常要靠 KLT 或内核可见线程

8. 简答题模板

题目：比较 KLT 与 ULT

内核级线程 KLT 由操作系统内核管理，内核能够直接感知和调度线程，因此一个线程阻塞时仍可调度同进程其他线程，并且在多处理器环境中可以实现真正并行，但线程管理和切换需要内核介入，开销较大。用户级线程 ULT 由用户空间线程库管理，线程切换不需要进入内核，开销较小，且调度灵活；但内核并不知道线程存在，因此一个线程阻塞会导致整个进程阻塞，也难以充分利用多处理器并行能力。

9. 考前速背版

  • KLT：内核管，能并行，阻塞不拖全进程，开销大
  • ULT：用户管，切换快，调度灵活，但一个阻塞全阻塞
  • ULT 适合逻辑并发
  • KLT 适合物理并行

2.4.3 多线程实现的混合策略

一句话理解

混合策略就是把 ULT 的轻量和 KLT 的并行性结合起来。

1. 基本思想

课件描述：

  • 线程创建在用户空间完成
  • 多个 ULT 映射到若干 KLT
  • 通过调整 KLT 数量，获得更好的并行效果

这本质上就是一种 M:N 映射模型。

2. 为什么要混合

纯 ULT 的问题：

  • 阻塞影响整个进程
  • 无法充分利用多核

纯 KLT 的问题：

  • 切换和管理成本较高

混合模型的目标：

  • 继承 ULT 的灵活和轻量
  • 结合 KLT 的内核并行支持

3. 主要特点

  • 线程创建在用户空间完成
  • 线程调度和同步可在应用层进行
  • 多个用户线程映射到少量或若干内核线程
  • 程序员/运行时可调整 KLT 数量以优化性能

4. 状态理解

课件给出：

  • KLT 有三态，系统调度负责
  • ULT 也有三态，用户调度负责
  • 活跃态 ULT 代表已绑定 KLT 的线程
  • 可借助 jacketing 和用户调度调整活跃线程

本质

这是一个“双层调度”：

  1. 用户级调度器先在 ULT 中选
  2. OS 再在 KLT 层面调度

  • 混合策略的基本思想
  • 为什么提出混合策略
  • 它解决了 ULT/KLT 的哪些问题
  • M:N 映射思想

7. 易错点

  1. 混合策略不是简单叠加，而是双层调度
  2. ULT 数一般多于或等于 KLT 数
  3. 线程创建常在用户空间，但真正并行能力依赖底层 KLT

8. 简答题模板

题目：简述多线程实现的混合策略

多线程实现的混合策略综合了用户级线程和内核级线程两种方法。在线程创建、部分调度和同步方面主要在用户空间完成，而多个用户级线程再映射到若干个内核级线程上，由内核对这些内核级线程进行调度。这样既保留了用户级线程切换开销小、调度灵活的优点，又利用了内核级线程可并行执行、阻塞影响较小的优势，从而在整体上获得更好的性能。

9. 考前速背版

  • 混合策略 = ULT + KLT
  • 本质：M:N 映射
  • 用户层轻量调度，内核层支持并行
  • 目标：结合优点、减少缺点

六、2.5 处理器调度

2.5.1 处理器调度的层次

一句话理解

调度不是只有“选谁上 CPU”这一层，而是从“让谁进系统”“谁留在内存”“谁此刻运行”三层共同完成。

1. 三个层次

课件给出：

  1. 高级调度（长程调度/作业调度）
  2. 中级调度（中程调度/平衡负载调度）
  3. 低级调度（短程调度/进程调度）

2. 各层次的作用

2.1 高级调度

决定：

  • 是否接受作业/用户连接
  • 是否让新建进程进入系统
  • 是否把作业从后备队列调入主存

批处理系统中

高级调度就是典型的作业调度。

分时系统中

也可以体现在是否接纳新任务。

2.2 中级调度

决定：

  • 哪些进程允许驻留主存
  • 哪些进程应被换出成为挂起态
  • 哪些挂起进程应恢复

目标

  • 提高内存利用率
  • 平衡系统负载
  • 提高吞吐量

2.3 低级调度

决定：

  • 就绪队列中哪个进程/线程下一刻占用 CPU

这是考试中“调度算法”大多实际针对的层次。

3. 三层关系图

flowchart TD
A[后备作业/新建进程] -->|高级调度| B[就绪/等待等在内存中的进程]
B -->|中级调度: 换入换出| C[挂起就绪/挂起等待]
C -->|中级调度恢复| B
B -->|低级调度| D[运行态]
D -->|阻塞/时间片到/结束| B

4. 低级调度的主要功能

课件列出：

  • 记住进程/线程状态
  • 决定什么时候获得处理器
  • 决定占用多长时间
  • 把处理器分派给谁
  • 收回处理器

本质

低级调度就是：

在所有可运行实体之间做瞬时 CPU 分配决策

5. 高频考点

  • 调度层次划分
  • 高级/中级/低级调度的区别
  • 中级调度与挂起的关系
  • 低级调度为什么是 OS 最核心部分

6. 易错点

  1. 低级调度才是真正“选谁上 CPU”
  2. 中级调度处理的是“换入换出、挂起恢复”
  3. 高级调度更偏向“是否让任务进入系统”
  4. 并非所有系统都同样强调三级调度，但概念上要会分

7. 简答题模板

题目：说明处理器调度的三个层次

处理器调度通常分为高级调度、中级调度和低级调度。高级调度又称长程调度或作业调度，主要决定哪些作业或新建进程可以进入系统并参与执行；中级调度又称中程调度，主要通过换入换出决定哪些进程驻留主存、哪些进程进入挂起状态，以平衡系统负载；低级调度又称短程调度或进程调度，主要从就绪进程或线程中选择一个分派处理器执行，是操作系统中最核心的调度层次。

8. 考前速背版

  • 高级：谁能进系统
  • 中级：谁留在内存
  • 低级：谁上 CPU
  • 低级调度最核心

2.5.2/2.5.3 处理器调度算法

一句话理解

调度算法就是 CPU 的分配规则，不同算法在公平性、响应速度、吞吐量和周转时间之间做不同权衡。

1. 选择调度算法的原则

课件给出五个核心指标：

指标 含义 希望目标
资源利用率 CPU/设备尽量忙 越高越好
响应时间 用户发请求到首次响应时间 越短越好
周转时间 提交到完成总时间 越短越好
吞吐量 单位时间完成进程数 越大越好
公平性 各进程获得合理份额 越公平越好

常见矛盾

  • 响应时间优先时，可能牺牲公平性
  • 吞吐量优先时，可能牺牲长作业体验
  • 公平性太强时，可能影响平均性能

2. 抢占式与非抢占式

非抢占式

当前运行进程除非：

  • 结束
  • 阻塞
  • 主动放弃

否则不会被强制剥夺 CPU。

抢占式

OS 可以因：

  • 时间片到
  • 更高优先级进程到达

强行收回 CPU。

对比

项目 非抢占式 抢占式
实现 简单 较复杂
响应性 差 好
适合 批处理 分时/实时
风险 长作业霸占 CPU 开销更大

3. 优先数调度算法

思想

根据优先级选择下一个运行进程。

类型

  • 抢占式优先级调度
  • 非抢占式优先级调度

优先级确定依据

  • 任务紧迫程度
  • 交互性
  • 外设使用频度
  • CPU 服务时间长短
  • 进入系统时间等

优点

  • 能体现任务轻重缓急

缺点

  • 可能造成低优先级进程饥饿

4. FCFS：先来先服务

思想

谁先进入就绪队列，谁先运行。

特点

  • 非抢占式
  • 实现简单
  • 按到达顺序服务

优点

  • 公平直观
  • 实现最简单

缺点

  • 短进程可能长时间等待
  • 有“护航效应”：I/O 型短任务被长 CPU 型任务拖住
  • 响应差

适用场景

  • 简单批处理
  • 对响应性要求不高的系统

5. SPN：最短进程优先（SJF 思想）

思想

选择估计运行时间最短的进程。

特点

  • 非抢占式
  • 有利于降低平均周转时间

优点

  • 对平均周转时间很有利

缺点

  • 需要预估执行时间
  • 长进程可能饥饿

6. SRT：最短剩余时间优先

思想

始终选择剩余执行时间最短的进程。

特点

  • 抢占式
  • 新到达的更短任务可抢占当前任务

优点

  • 对平均完成时间/周转时间通常更优

缺点

  • 更依赖精确预测
  • 长作业更容易被饿死
  • 抢占频繁带来开销

7. HRRN：最高响应比优先

思想

选择响应比最高的进程。

正确公式

更标准的写法是：

$$
响应比 = \frac{等待时间 + 要求服务时间}{要求服务时间}
= 1 + \frac{等待时间}{要求服务时间}
$$

说明：课件部分位置把“响应比”写成 等待时间 / 估计计算时间，这不够标准。标准教材中的 HRRN 公式应为上式。

为什么这个算法好？

  • 短进程初始响应比高
  • 长进程随着等待时间增长，响应比也会提高
  • 因而比 SJF 更不容易饿死长进程

特点

  • 非抢占式
  • 综合考虑等待时间和服务时间

8. RR：时间片轮转

思想

就绪进程按顺序轮流使用 CPU，每次只用一个时间片。

过程

  • 时钟周期性中断
  • 时间片到，当前进程被放回就绪队列尾部
  • 选择下一个进程运行

优点

  • 公平
  • 响应好
  • 适合交互式系统

缺点

  • 时间片太大：退化成 FCFS
  • 时间片太小：切换开销大

时间片选择原则

  • 过长：交互性差
  • 过短：系统忙于切换
  • 一般应在“响应性”和“开销”间折中

9. 多级反馈队列（Feedback）

思想

建立多个优先级不同的就绪队列：

  • 高优先级队列时间片短
  • 低优先级队列时间片长
  • 进程会根据执行行为在不同队列间移动

为什么叫“反馈”

因为系统根据进程“表现”动态调整其优先级：

  • 用不完时间片、经常阻塞：说明交互性强，优先级可高
  • 老是用满时间片：说明偏 CPU 型，优先级可逐渐降低

典型规则（常见教材版）

  1. 新进程先进入最高优先级队列
  2. 若时间片用完未完成，则降到下一队列
  3. 若阻塞后很快返回，可能保留较高优先级
  4. 调度时总优先选择高优先级队列

优点

  • 兼顾交互作业和计算作业
  • 响应快
  • 灵活
  • 现代 OS 广泛采用类似思想

缺点

  • 实现复杂
  • 参数设置复杂
  • 可能产生饥饿，需要老化等机制修正

10. 彩票调度

思想

给进程分配若干“彩票”，调度时随机抽一张，持有该彩票的进程获得资源。

含义

  • 彩票越多，获得 CPU 的概率越大
  • 是一种概率公平机制

特点

  • 灵活
  • 可表达资源份额
  • 适合研究与特定系统设计

15. 易错点

  1. RR 是抢占式，因为靠时钟中断收回 CPU
  2. SPN 非抢占，SRT 抢占
  3. HRRN 公式不是单纯等待时间除服务时间，而是 (等待+服务)/服务
  4. 多级反馈队列并不是简单多个 RR 队列，而是带优先级反馈机制
  5. FCFS 看起来公平，但对短任务不友好

16. 简答题模板

题目：简述时间片轮转调度算法及其特点

时间片轮转调度算法是一种抢占式调度算法。系统按照进程进入就绪队列的先后顺序，轮流把 CPU 分配给各进程，每次只允许进程运行一个时间片。当时间片用完时，由时钟中断收回处理器，并把该进程送回就绪队列尾部，再调度下一个进程。该算法实现简单、公平性较好、响应时间短，适合分时系统；但如果时间片过长会退化为先来先服务，过短则会导致频繁切换、系统开销增大。

题目：简述多级反馈队列调度算法思想

多级反馈队列调度算法把就绪进程按优先级组织成多个队列，高优先级队列分配较短时间片，低优先级队列分配较长时间片。新到达进程通常先进入高优先级队列，若其时间片用完仍未完成，则被降到较低优先级队列；调度时始终优先选择高优先级队列中的进程。该算法能够根据进程执行行为动态调整优先级，兼顾交互型和计算型作业，是现代操作系统中常见的调度思想。

17. 考前速背版

  • 指标：利用率、响应时间、周转时间、吞吐量、公平性

  • FCFS：先来先服务，非抢占

  • SPN：最短进程优先，非抢占

  • SRT：最短剩余时间优先，抢占

  • HRRN：最高响应比优先，非抢占
    $$
    HRRN = \frac{等待+服务}{服务}
    $$

  • RR：时间片轮转，抢占

  • Feedback：多级反馈，抢占，现代常用

七、全章高频考点总汇

高频考点 1：PSW、PCB、进程映像、上下文的区别

名词 定义关键词
PSW 程序状态字，反映 CPU 当前状态
PCB 进程控制块，OS 管理进程的数据结构
进程映像 PCB + 程序 + 数据 + 栈等进程整体实体
上下文 进程恢复执行所需的环境与状态

高频考点 2：中断、异常、系统调用区别

  • 中断：外部、异步
  • 异常：内部、同步、由当前指令引起
  • 系统调用：程序主动陷入内核请求服务

高频考点 3：三态模型与挂起模型

  • 三态：运行、就绪、等待
  • 扩展：挂起就绪、挂起等待
  • 不能直接：
    • 等待 → 运行
    • 就绪 → 等待

高频考点 4：模式切换 vs 进程切换

  • 模式切换：权限变化
  • 进程切换：执行实体变化
  • 进程切换一定包含模式切换
  • 模式切换不一定包含进程切换

高频考点 5：线程 vs 进程

  • 进程：资源分配单位
  • 线程：调度单位
  • 同进程线程共享资源，不共享寄存器/栈

高频考点 6：KLT vs ULT

  • KLT：内核可见、能并行、切换较重
  • ULT：用户可见、切换轻、阻塞拖整个进程

高频考点 7：调度算法对比

最爱考：

  • RR
  • SPN
  • SRT
  • HRRN
  • 多级反馈队列

九、考前速背版

1. 处理器与模式

  • PC：下一条指令地址
  • IR：当前指令
  • CC：结果标志
  • PSW：程序状态字
  • 指令分特权/非特权
  • 模式分内核态/用户态

2. 中断

  • OS 是中断驱动的
  • 中断：外部异步
  • 异常：内部同步
  • 系统调用：主动陷入
  • 中断处理后可能回原进程，也可能换进程

3. 进程

  • 进程 = 程序的一次运行活动
  • 进程 = 资源分配和调度单位
  • 三态：运行、就绪、等待
  • 扩展态：挂起就绪、挂起等待
  • 等待不能直接到运行

4. PCB / 映像 / 上下文

  • PCB：进程档案
  • 进程映像：PCB + 程序 + 数据 + 栈
  • 上下文：恢复执行所需环境

5. 原语与切换

  • 原语：不可分割、原子性
  • 进程切换：保存旧上下文 + 调度 + 恢复新上下文
  • 模式切换：用户态 ↔ 内核态
  • 进程切换一定包含模式切换

6. 线程

  • 进程管资源，线程管执行
  • 线程是调度单位
  • 线程共享进程资源
  • 各线程有自己的栈和寄存器

7. KLT / ULT

  • KLT：内核管理，能多核并行，切换开销大
  • ULT：用户管理，切换快，但一个阻塞全进程阻塞
  • 混合策略：M:N，结合二者优点

8. 调度算法

  • FCFS：先来先服务，非抢占
  • SPN：最短进程优先，非抢占
  • SRT：最短剩余时间优先，抢占
  • HRRN：最高响应比优先，非抢占
  • RR：时间片轮转，抢占
  • Feedback：多级反馈，抢占

必背公式

$$
周转时间 = 完成时间 - 到达时间
$$

$$
带权周转时间 = \frac{周转时间}{服务时间}
$$

$$
HRRN = \frac{等待时间 + 服务时间}{服务时间}
$$
